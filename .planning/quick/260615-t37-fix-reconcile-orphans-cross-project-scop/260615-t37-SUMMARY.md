# Quick Task 260615-t37 — Summary

## Problem

`_reconcile_orphans` (`src/arduis/window.py`, the `_on_ls` callback ~line 1914) built its
`live` set from `self._store.all()`, which only covers the **active** project's tasks. A
background project with an enabled compose stack was therefore flagged as an orphaned
`arduis-*` stack at startup, producing a false-positive toast. No teardown / data /
lifecycle impact — purely cosmetic. Flagged by the v1.0 milestone audit
(`gaps.integration[reconcile-orphans-scope]`, severity cosmetic).

## Fix

The `live` comprehension now unions tasks across **every** registered project, mirroring the
canonical `_all_tasks()` pattern (window.py:2022):

```python
live = {
    compose.sanitize_project_name(t.branch)
    for p in self._registry.all()
    for t in p.store.all()
}
```

No other logic in `_on_ls` / `_reconcile_orphans` was touched.

## Test

Added `test_reconcile_orphans_cross_project` to `tests/test_window_projects.py`. It drives the
real `_reconcile_orphans` (stubbed `run_compose_async` + captured toast) and asserts that a
background project's stack is **not** flagged as an orphan while a genuinely unmatched stack
still is.
