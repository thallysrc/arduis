# Phase 03.4 Plan 04: Multi-Project Lifecycle Correctness Summary

Closed the three lifecycle gaps the project switcher opened, each fixed at the real call site
RESEARCH identified: the new-task/resume cap gate now counts the UNION of every open project's
agents (D-09); removing a project tears down its live tasks behind a DESTRUCTIVE confirm and drops
it from the list without ever touching disk (D-10); and app-exit iterates `registry.all()` so every
project's agents and containers die with that project's OWN compose identity — no cross-project
orphans (D-11). Three atomic commits; `main` stays launchable and the full suite is green (426).

## What Was Built

- **Task 1 — cap-union gate (D-09).** Added `_all_tasks()` returning
  `[t for p in self._registry.all() for t in p.store.all()]`. Replaced both
  `caps.at_cap(self._store.all())` call sites (the new-task gate `_on_new_worktree_clicked` and the
  resume gate `_resume_gated`) with `caps.at_cap(self._all_tasks())`, and made the
  `_prompt_hibernate_then` heading count use `caps.active_count(self._all_tasks())`. The chooser LIST
  stays scoped to the active project's active tasks (documented deferral, RESEARCH Open Q2). The cap
  is now machine-global across projects (RAM is a machine resource).

- **Task 2 — per-project remove with teardown (D-10).** Added a right-click `Gtk.GestureClick`
  (secondary button) on each project tab popping a `Gio.Menu` with "Remover projeto" targeting the
  new `win.remove_project(root)` `Gio.SimpleAction` (registered in `_install_row_actions`, mirroring
  `win.switch_project`). `_remove_project(root)`: if the project has NO live (ACTIVE) tasks it drops
  silently (no dialog); if it has live tasks it presents a DESTRUCTIVE `Adw.AlertDialog`
  (`set_response_appearance("remove", DESTRUCTIVE)`, `set_default_response("cancel")`, UI-SPEC copy)
  and, on confirm, kills each live task's terminal groups + clears its state files, runs
  `_teardown_project_containers(proj)` (per-project `compose down -v`), then `_drop_project(root)`.
  Added `_project_isolation_available(proj)` (per-project variant of `_isolation_available`),
  `_teardown_project_containers(proj)` (brief-sync `compose down -v` per enabled task using THIS
  project's compose identity, argv built as a LIST via `compose.down_argv` + `wrap_argv`), and
  `_drop_project(root)` (registry.remove + projects.json rewrite + active-reselect). NO disk-dir
  deletion anywhere — the teardown is kill pgids + compose down -v ONLY (D-10 hard rule).

- **Task 3 — app-exit all-projects teardown (D-11) + regression.** Rewrote `_on_close_request` so
  BOTH the killpg/state-file teardown and the container-down loop iterate `self._registry.all()`
  (outer loop), reusing `_teardown_project_containers` so each project's containers go down with that
  project's OWN `compose_path`/`container_state`/`project_name` — never the active project's name for
  all. Extended `tests/test_window_projects.py` with four headless cases (bare-`__new__` + monkeypatch
  stub, no display): `test_cap_gate_counts_union` (7-task union > cap → prompt branch),
  `test_remove_with_live_tasks_tears_down_then_drops` (teardown + compose-down + registry drop, no
  disk deletion), `test_remove_no_live_tasks_is_silent` (HIBERNATED-only → no dialog, no teardown),
  and `test_close_request_tears_down_all_projects` (2 `_teardown_session_terminals` calls + 2 DISTINCT
  per-project compose-down argv — a name-reuse bug fails this).

## Deviations from Plan

None of substance — plan executed as written. One implementation refinement worth noting:

**[Rule 1 - simplification] App-exit container loop reuses `_teardown_project_containers`.**
- **Found during:** Task 3.
- **Detail:** The plan described inlining the per-project compose-down loop into `_on_close_request`.
  Instead the exit handler calls the `_teardown_project_containers(proj)` helper introduced in Task 2
  (per-project compose identity, argv list, brief-sync `subprocess.run`). This guarantees remove
  (D-10) and exit (D-11) use the IDENTICAL per-project compose path — one place to be correct, so the
  D-11 "two distinct per-project argv" property can never drift from the remove path. Behavior matches
  the plan exactly (each project's OWN identity, list argv, timeout=10, swallowed errors).

## Threat Mitigations Applied

- **T-03.4-11** (Tampering, compose down argv): `_teardown_project_containers` builds the down argv
  via `compose.down_argv(project_name, task_dir)` (a LIST) wrapped by `HostRunner.wrap_argv` — never a
  shell string — so a crafted project/branch name cannot inject. Applied at both the remove (D-10) and
  app-exit (D-11) call sites.
- **T-03.4-12** (Tampering, remove disk side-effects): `_remove_project` / `_drop_project` only kill
  pgids + clear state files + `compose down -v` + drop the registry/JSON entry. NO `git worktree
  remove`, NO `rm`, NO dir delete anywhere. `test_remove_with_live_tasks_tears_down_then_drops` asserts
  the registry drop occurred with no deletion call (grep confirms none exist in the method).
- **T-03.4-13** (resource leak / no-orphans): `_on_close_request` outer-loops `registry.all()` for both
  killpg and compose-down, each with the project's OWN compose identity;
  `test_close_request_tears_down_all_projects` asserts 2 distinct per-project teardown + compose calls.
- **T-03.4-14** (resource leak / global cap): `_all_tasks()` feeds the union to `caps.at_cap` at every
  gate; `test_cap_gate_counts_union` pins it (7 > cap → prompt branch).
