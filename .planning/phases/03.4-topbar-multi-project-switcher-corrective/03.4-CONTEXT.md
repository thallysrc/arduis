# Phase 03.4: Topbar = multi-PROJECT switcher (corrective) - Context

**Gathered:** 2026-06-15 (--auto mode; PO pre-decided the core model, Claude picked the sub-defaults)
**Status:** Ready for planning

<domain>
## Phase Boundary

Correct the topbar to its right LEVEL in the approved 3-level model (`docs/MOTIVATION.md`:
"Topbar = projetos. Um projeto é uma pasta raiz multi-repo"). The topbar becomes a switchable
list of **PROJECTS** — each a whole multi-repo root (`Livon-Saude`, `KarveLabs`) — added via an
"Open project" picker and **remembered across launches**, with multiple projects allowed to run
**in parallel ("both alive")**. Selecting a project swaps the sidebar (its tasks) + workspace.
Member repos (`backend`/`frontend`/`keycloak`) leave the topbar and surface only in the "Nova
task" dialog.

**Supersedes:** Phase 03.3 (topbar repo chips) and Phase 03.2 D-06 (topbar = name only) IN FULL —
both modeled the topbar at the wrong level.
**Out of scope:** the Command/Swarm/Review room tabs from the mockup (v2); per-project attention
status rollup on the tab (defer — Phase 4 attention stays per-task); any change to the task model
itself (task = worktrees of a project's chosen repos — unchanged from 03.2).
</domain>

<decisions>
## Implementation Decisions

### Project model & state architecture
- **D-01:** Introduce a GTK-free `Project` object encapsulating its own `root`, `member_repos`,
  task store (`SessionStore`), and live terminals/pgids. A `ProjectRegistry` (also GTK-free) holds
  the open projects + the active-project pointer. This replaces the `window.py` singletons
  (`_project_root`, `_member_repos`, `_repo_root`, single `_store`). Keep GTK out of these (same
  discipline as the existing `SessionStore` / `topbar.py`).
- **D-02:** Each project owns its OWN `SessionStore` (its tasks). The window renders the
  active project's store; switching projects re-points the rendered store.

### Topbar switcher widget
- **D-03:** The topbar renders one **tab-style button per open project** (project basename as
  label), in the headerbar at the existing `pack_start` slot now used by the chip bar
  (`_chip_bar`). Selecting a tab makes that project active. A trailing **"+ Open project"** button
  triggers a `Gtk.FileDialog` folder picker. Few projects expected → tabs (not a dropdown); no
  horizontal-scroll gymnastics (an overflow menu only if it ever exceeds the header width, reusing
  the 03.3 `+N` overflow pattern).
- **D-04:** The per-repo chip rendering from 03.3 (`_make_chip`, `_restyle_chips`,
  `_reflect_active_chips`, `topbar.py ChipState`) is **removed/repurposed** — the topbar no longer
  carries repo chips. The chip-bar Box slot is reused for the project tabs.

### Project-list persistence
- **D-05:** Persist the remembered-projects list to a **JSON state file** at the XDG config dir
  (`$XDG_CONFIG_HOME/arduis/projects.json`, fallback `~/.config/arduis/projects.json`) using the
  stdlib `json` module. Rationale: `.arduis.toml` stays **read-only** per CLAUDE.md (`tomllib`),
  and this avoids adding the optional `tomli-w` dependency. This is an APP-STATE file (distinct
  from the "disk is source of truth for tasks" convention — task discovery via `_scan_tasks`
  stays per-project on disk; only the *set of project roots* is persisted, since arbitrary roots
  can't be rediscovered).
- **D-06:** Persisted per project: the absolute `root` path. Also persist `last_active_project`
  so relaunch restores focus. Validate each root on load (skip/flag missing dirs gracefully).

### Launch behavior / backward-compat
- **D-07:** On launch, resolve the cwd's project (existing `_resolve_project` logic / D-07
  "launch lands you in it"), **auto-register it** into the remembered list if absent, and select
  it as active. Preserves today's single-project launch behavior as the degenerate case.

### "Both alive" parallelism & resource caps
- **D-08:** Switching projects does NOT tear down the other projects' terminals/agents/containers
  — they keep running (PO decision). Background projects' VTE widgets are **retained in memory**
  (detached from the visible canvas, not destroyed) so switch-back is instant and state-preserving.
- **D-09:** The existing configurable active agent/container cap (RAM-03) stays **GLOBAL across
  all projects** — it counts live agents/containers machine-wide regardless of project, since RAM
  is a machine-global resource. Per-task hibernation (RAM) continues to apply within each project.

### Add / remove lifecycle
- **D-10:** Removing a project from the topbar **tears down its live tasks** (kill agent pgids +
  `docker compose ... down -v` for any isolated tasks), honoring the "teardown garantido"
  requirement. If the project has live/running tasks, confirm before removing. Removing only drops
  it from the remembered list + tears down — it never deletes worktrees/task dirs on disk.
- **D-11:** App-exit teardown must iterate **all** open projects (no orphans across projects), not
  just the active one — extend the existing window-close teardown.

### Member-repo surfacing
- **D-12:** `project.detect_member_repos` + the D-04/03.3 `.git`-directory filter **survives
  unchanged** and now feeds the "Nova task" dialog's per-repo picker for the ACTIVE project. The
  topbar no longer surfaces repos at all.

### Claude's Discretion
- Exact tab widget (Gtk.ToggleButton group vs `Adw.ViewSwitcher`-style) and active-tab styling.
- Canvas widget retain/detach mechanism for background projects (e.g. one `Gtk.Stack` page per
  project vs manual unparent/reparent) — planner/researcher to choose the cleanest VTE-safe path.
- JSON schema details + atomic-write strategy for `projects.json`.
- Whether the "+ Open project" picker validates that the chosen dir is a plausible project root.
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/arduis/project.py::detect_member_repos` + `.git`-directory filter — feeds the New-task
  dialog repo picker for the active project (D-12). Unchanged.
- `src/arduis/SessionStore` (GTK-free task store) — becomes per-`Project` (D-02). The model is
  already GTK-free, so wrapping N of them behind a registry is natural.
- 03.3 `+N` overflow menu pattern (`win.toggle_chip` style action) — reuse shape only if project
  tabs ever overflow the header (D-03).
- `_chip_bar` Box at `header.pack_start` — the physical slot reused for project tabs (D-03/D-04).

### Established Patterns
- `window.py` currently bakes ONE project as singletons: `_project_root`, `_member_repos`,
  `_repo_root`, single `_store`, `_resolve_project()` run once at startup, `_scan_tasks()` rooted
  at the single project. This phase generalizes these to the active `Project` via the registry.
- Window-close teardown (SIGHUP→SIGKILL pgids; compose down) — extend to iterate ALL projects
  (D-11). `_on_resume`/hibernate semantics stay per-task within a project.
- Workspace-swap funnel (`_reflect_active_chips` callers, `_MAIN_SID` pinned leaf) — project
  switch is a higher-level swap that re-points the active store, then runs the existing
  task→workspace swap for that project's last-active task.

### Integration Points
- New: `projects.json` read at startup (seed registry + restore `last_active_project`) and
  written on add/remove/switch.
- New: "+ Open project" → `Gtk.FileDialog` (folder select) → `detect_member_repos` → register.
- Modified: `_resolve_project` becomes "resolve cwd → Project, register, select" (D-07).
- Removed: `topbar.py ChipState` + the chip render/restyle/reflect methods in `window.py`.
</code_context>

<specifics>
## Specific Ideas

- PO's concrete example: topbar shows `Livon-Saude` and `KarveLabs` side by side; each is a whole
  multi-repo; the user works on both in parallel and switches between them on the topbar.
- "Both alive" is explicit and intentional: switching away must NOT kill the other project's
  agents/containers (state preserved) — a deliberate RAM tradeoff bounded by the global cap (D-09).
</specifics>
