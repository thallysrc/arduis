# Phase 03.4 Plan 03: Multi-Project Switcher Structural Lift Summary

The largest plan: generalized `window.py`'s one-project singletons onto a `ProjectRegistry`,
partitioned the GTK widget maps per project so background projects stay "both alive" on switch,
namespaced every status-file path per project (A3), wired the `projects.json` launch sequence
with cwd auto-register (D-07), and rendered the topbar as switchable project tabs + a
"+ Abrir projeto" folder picker. `main` stays launchable after every task (5 atomic commits);
teardown/remove/cap-union is deliberately left to Plan 04.

## What Was Built

- **Task 1a — registry + accessor delegation.** Added `self._registry = ProjectRegistry()` + a
  per-window bootstrap `Project` fallback. The active-state singletons (`_store`,
  `_project_root`, `_project_name`, `_member_repos`, `_compose_path`, `_container_state`, and the
  `_repo_root`/`_repo_name` aliases) became read/write `@property` delegations onto
  `registry.active()` (bootstrap fallback when no project is selected). `_docker_available` stays a
  plain machine-fact field. With exactly one project, behavior is unchanged; the bare-`__new__`
  conclude test still assigns `win._store` (the setter writes onto the bootstrap).
- **Task 1b — per-project GTK map partition (D-08).** Moved `_leaf_by_sid`, `_term_by_sid`,
  `_layouts`, `_active_workspace_sid`, and the Phase-4/5 `dot/pane_dot/notif/subline/record/calm/
  activity/activity_last_handled/badge` maps onto a lazily-created per-`Project` bundle
  (`_gtk_maps`), exposed via `@property` delegations to the active project's bundle (`_m()`).
  `_reflect_layout`'s existing unparent loop (`for leaf in self._leaf_by_sid.values()`) now only
  touches the ACTIVE project's leaves — background projects' leaves sit unparented but ALIVE.
  `project.py` stays gi-free (the bundle holds GTK values but is plain dicts). `_sid_by_row`/
  `_row_by_sid`/`_footer_label` stay window-global (they describe the single rendered sidebar).
- **Task 1c — status-file namespacing (A3).** Added `_proj_term_id(term_id)` wrapping the term_id
  with the active project's `project_term_id`. Applied at all 4 status-file derive sites — the
  spawn env path + `_record_by_state_file` write key, the watcher-tick re-read lookup, and both
  teardown unlink loops — so the path WRITTEN at spawn equals the path READ by the watcher/tick/
  teardown. Two projects with same-named branches never share a status file.
- **Task 2 — launch sequence (D-05/D-06/D-07).** Added `self._projects_json` (XDG) +
  `_init_projects()`: `load_projects` → for each remembered root `ensure_project` + compute
  compose_path + `_scan_tasks(project=...)` into ITS own store (D-02); auto-register the cwd if
  absent (D-07); choose active (cwd wins when inside a project, else last_active, else None);
  persist; render active sidebar + project tabs + reconcile orphans. Split `_resolve_project` into
  pure `_resolve_cwd_project` + `_detect_compose_path` + `_refresh_project_chrome`, and
  parameterized `_scan_tasks(project=...)` to seed a background project's store without touching
  the active-project UI. Startup now calls `_init_projects()` in place of the
  `_resolve_project()` + `_scan_tasks()` pair.
- **Task 3 — the switcher (D-03/D-08).** `_build_project_tabs` renders one flat linked
  `Gtk.ToggleButton` per project (label via `set_text` ONLY, T-03.4-07; tooltip = full root path),
  the active tab carrying `.arduis-chip-active` + a semibold (`.arduis-tab-label-active`, 600)
  label, a `+N` overflow `MenuButton` targeting `win.switch_project(root)` past
  `_MAX_VISIBLE_CHIPS`, and a trailing "+ Abrir projeto" flat button. The picker is
  `Gtk.FileDialog.select_folder` → `_open_project` (detect members + compose + scan + persist +
  switch). `_switch_project` remembers the outgoing workspace, `set_active`, rebuilds the sidebar,
  reuses `_swap_workspace` (re-grabs focus, Pitfall 2), restyles the tabs, and persists
  last_active — and NEVER tears down the previous project (D-08). Registered the
  `win.switch_project` action and added the `.arduis-tab-label-active` CSS.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Active-project choice: cwd wins over last_active when launched inside a project**
- **Found during:** Task 2 (the D-07 behavioral test failed against the plan's literal step-d order)
- **Issue:** The plan's step d ("`last` if still registered, else cwd") contradicted the must-have
  truth "On launch the cwd project auto-registers and **is selected**" and the required test
  asserting `active().root == rootA` when the cwd was absent from the persisted list. Following the
  literal order selected the persisted `last_active` (rootB) over the freshly auto-registered cwd.
- **Fix:** Reordered the choice to "cwd_root if it resolved (you always land in the project you
  launched in), else last_active from a neutral launch dir, else None." This satisfies D-07 and the
  required behavioral test; `last_active` is still honored when launched from a non-project dir.
- **Files modified:** src/arduis/window.py (`_init_projects` step d)
- **Commit:** 486b9e2

Otherwise the plan executed as written.

## Threat Mitigations Applied

- **T-03.4-07** (Tampering, project tab labels): tab labels are `Gtk.Label.set_text(project.name)`
  only — never `set_markup` on a filesystem-derived basename (the `+N` overflow uses
  `Gio.MenuItem.new(proj.name, None)`, also plain text).
- **T-03.4-08** (Tampering, scan on a chosen root): `_open_project` feeds the chosen root only to
  `detect_member_repos` / `_scan_tasks`, which reach git argv as discrete list elements via the
  existing HostRunner discipline — no shell strings introduced.
- **T-03.4-09** (Spoofing/Elevation, "+ Abrir projeto" arbitrary dir): accepted — same trust model
  as today's launch dir (the user chose it; arduis only reads git/scan).
- **T-03.4-17** (Spoofing-of-state, cross-project status collision): `_proj_term_id` namespaces the
  term_id at all 4 status-file call sites + the watcher key; the path WRITTEN == path READ, so two
  projects' same-named branches never share a status file (A3).
