# Phase 03.4: Topbar = multi-PROJECT switcher (corrective) - Research

**Researched:** 2026-06-15
**Domain:** GTK4/libadwaita widget-lifecycle (VTE retention across switch), Python app-state architecture refactor (singletons ‚Üí registry), XDG JSON persistence
**Confidence:** HIGH (codebase-grounded; the load-bearing VTE-unparent claim is already proven in the existing code)

## Summary

This is an **architectural refactor**, not a UI tweak. `window.py` bakes "one arduis = one
project" as instance singletons (`_project_root`, `_member_repos`, `_repo_root`, `_repo_name`,
one `_store`/`SessionStore`, plus all the Phase-7 container state and Phase-4 attention maps).
The phase generalizes those into a GTK-free `Project` object (own root, member_repos, own
`SessionStore`, own live terminals/pgids) held behind a GTK-free `ProjectRegistry` (open
projects + active pointer). The topbar's 03.3 repo-chip render is deleted and the same
`_chip_bar` Box slot is reused to render one tab button per open project, plus a "+ Open
project" `Gtk.FileDialog` folder picker. The remembered-projects set is persisted to
`$XDG_CONFIG_HOME/arduis/projects.json` (stdlib `json`, atomic temp+rename ‚Äî `tomllib` is
read-only per CLAUDE.md so TOML is not an option for writes).

The single highest-risk unknown ‚Äî "can background projects' VTE terminals stay alive while
hidden?" ‚Äî is **already answered in the existing code**: `_reflect_layout` (window.py:3811)
unparents every mapped leaf and re-hangs only the active subset, and its own comment states
*"VTE widgets + their PTY children SURVIVE unparenting (A1/Pitfall 2) ‚Äî a swap never
respawns."* The workspace-swap funnel (`_swap_workspace`) already keeps N hidden workspaces'
terminals alive within a single project. "Both alive" across projects is the **same mechanism
one level up**: keep each project's `_leaf_by_sid`/`_term_by_sid`/`_layouts` maps alive; only
re-reflect the active project's tree into the shared `_canvas_slot`. The cleanest detach
mechanism therefore is **the existing unparent-on-reflect pattern, scoped per project** ‚Äî NOT a
`Gtk.Stack` (which would force every project's whole canvas subtree to be simultaneously
parented and is harder to reconcile with the per-workspace `_canvas_slot.set_child` funnel).

**Primary recommendation:** Introduce `Project` + `ProjectRegistry` as pure (GTK-free) types
that *own* the existing GTK-free state (SessionStore, container state dict, member repos, root).
Keep all GTK widget maps in `window.py` but **partition them per project** (a dict keyed by
project id, or moved onto the `Project` as plain attributes that window.py reads). The active
project's maps feed the existing reflect/swap/teardown funnels verbatim. Switching a project =
"re-point `self` accessors at the active Project, then run the existing task‚Üíworkspace swap for
that project's last-active task." Persist roots+last_active to `projects.json`. Reuse the exact
atomic-write pattern already in `appconfig.write_theme` (mkstemp in same dir ‚Üí `os.replace`).

## Architecture Patterns

### Recommended Project Structure
```
src/arduis/
‚îú‚îÄ‚îÄ project.py        # SURVIVES: detect_member_repos (D-12). ADD: Project, ProjectRegistry (GTK-free)
‚îú‚îÄ‚îÄ projects_store.py  # NEW (optional split): load/save projects.json (atomic). GTK-free, stdlib json
‚îú‚îÄ‚îÄ session.py        # SURVIVES unchanged: SessionStore is now OWNED per-Project
‚îú‚îÄ‚îÄ topbar.py         # ChipState DELETED (D-04). (file may be emptied/removed ‚Äî see "What to delete")
‚îî‚îÄ‚îÄ window.py         # renders the active Project; per-project widget maps; tab bar replaces chip bar
```
Placing `Project`/`ProjectRegistry` in `project.py` keeps the GTK-free discovery + model
together (one import surface). Splitting `projects.json` I/O into its own `projects_store.py`
keeps the persistence unit-testable in isolation (no GTK, no registry) ‚Äî recommended.

### Pattern 1: GTK-free Project owns the per-project state
```python
# Source: derived from existing SessionStore (session.py) + window.py singletons
# GTK-free ‚Äî imports NO gi.
@dataclass
class Project:
    root: str                       # absolute project root (was window._project_root)
    name: str                       # os.path.basename(root) (was _project_name)
    member_repos: list[str]         # detect_member_repos(root) (was _member_repos)
    store: SessionStore             # this project's tasks (was the single _store) ‚Äî D-02
    last_active_task: str | None = None   # task_id or _MAIN_SID, for switch-back
    # container availability is per-project (root compose may differ) ‚Äî was
    # _compose_path/_docker_available; container_state dict moves here too.
    compose_path: str | None = None
    container_state: dict = field(default_factory=dict)

class ProjectRegistry:
    """Open projects + the active pointer. GTK-free, in-memory; persisted set is the roots."""
    def __init__(self) -> None:
        self._projects: dict[str, Project] = {}   # keyed by root (stable, unique)
        self._active: str | None = None
    def add(self, p: Project) -> None: ...
    def get(self, root: str) -> Project | None: ...
    def all(self) -> list[Project]: ...
    def active(self) -> Project | None: ...
    def set_active(self, root: str) -> None: ...
    def remove(self, root: str) -> None: ...   # in-memory only; teardown is window.py's job
```
**When to use:** This is the spine of the phase. `window.py` reads `registry.active()` wherever
it previously read `self._project_root`/`self._member_repos`/`self._store`.

### Pattern 2: Per-project GTK widget maps (the "both alive" retention)
The GTK leaf/terminal/layout maps must NOT be a single set anymore ‚Äî each project keeps its own
so switching does not destroy the other's terminals. Two viable shapes (Claude's discretion):
- **(A) Move the maps onto `Project`** as plain (non-GTK-typed) dict attributes that window.py
  populates: `proj.leaf_by_sid`, `proj.term_by_sid`, `proj.layouts`, `proj.active_workspace_sid`,
  plus the Phase-4 dot/notif maps and Phase-7 compose-busy sets. (Keeps `Project` technically
  GTK-free ‚Äî the dicts just *hold* GTK objects window.py created.)
- **(B) Keep the maps in window.py but key them by project**: `self._leaf_by_sid_by_proj[root]`.
  More churn at every call site.

**Recommend (A) variant but keep the GTK *types* out of `project.py` annotations** (annotate as
`dict` / `object`), so `project.py` still imports no `gi` and stays unit-testable. window.py is
the only module that puts `Gtk.Widget`/`Vte.Terminal` *values* into those dicts.

```python
# Source: window.py:3811 _reflect_layout ‚Äî ALREADY unparents+reattaches; VTE survives.
# "both alive" = same pattern, but the leaves of NON-active projects are simply
# never re-hung (they sit unparented, fully alive, PTY children running).
def _reflect_layout(self) -> None:
    if self._canvas_slot.get_child() is not None:
        self._canvas_slot.set_child(None)
    for leaf in self._active_proj_leaves():   # ONLY the active project's leaves
        if leaf.get_parent() is not None:
            leaf.unparent()
    ...
```

### Pattern 3: Project switch reuses the workspace-swap funnel
Switching project = re-point window.py's "active" accessors at the new `Project`, rebuild the
sidebar from *that* project's `store`, then run the **existing** `_swap_workspace(sid)` for the
project's `last_active_task` (or `_MAIN_SID`). Persist `last_active_project`. The existing
`_swap_workspace` ‚Üí `_reflect_layout` chain handles canvas + focus; you are layering a
project-level swap on top of the proven task-level swap.

### Anti-Patterns to Avoid
- **Destroying background terminals on switch.** Violates D-08. Never call `_teardown_*` on a
  project switch ‚Äî only on remove (D-10) or app-exit (D-11).
- **One global `Gtk.Stack` of full canvases.** Re-plumbs the canvas funnel for no benefit; see
  Alternatives.
- **A second event loop / threads for "parallel" projects.** "Both alive" is purely widget
  retention + the existing per-PTY `spawn_async`; there is no new concurrency. Keep using
  `Gio.Subprocess`/`run_git_async` for non-PTY work [CITED: CLAUDE.md].
- **Re-`detect_member_repos` on every switch.** Compute once at open and store on the `Project`.

## Common Pitfalls

### Pitfall 1: term_id / task_id collisions across projects
**What goes wrong:** `task_id` defaults to the branch name (session.py:144); `term_id` is
`{task_id}:tN`. Two projects each with a `feature-x` task produce identical ids ‚Üí the
`_term_by_sid`, `_record_by_state_file`, and Phase-4 status-file paths collide.
**Why it happens:** ids were globally-unique only because there was one project.
**How to avoid:** Partition the GTK maps per project (Pattern 2) so a `feat:t0` in project A and
in project B are distinct dict entries under distinct projects. For the **status-file path**
(`attention.state_file_path(status_dir, term_id)`) the path is global ‚Üí either namespace the
status dir per project or prefix term_id. Flag explicitly for the planner.
**Warning signs:** opening a second project that shares a branch name with the first; status
dots flipping on the wrong project's task.

### Pitfall 2: VTE focus after re-attach
**What goes wrong:** after a project switch (unparent old, reflect new), keystrokes go nowhere.
**Why it happens:** GTK4 focus is lost when a widget is unparented.
**How to avoid:** the existing `_swap_workspace` already calls `term.grab_focus()` on the
swapped-in focused terminal (window.py:1772). Route project switch through the same funnel so
the grab happens. Re-grab the active project's focused terminal after switch.
**Warning signs:** typing does nothing right after clicking a project tab.

### Pitfall 3: app-exit teardown only iterates the active project
**What goes wrong:** `_on_close_request` (window.py:4294) iterates `self._store.all()` ‚Äî the
active project's store only. Background projects' agents/containers become orphans (violates
D-11).
**How to avoid:** change the close handler to iterate `registry.all()` ‚Üí for each project run
`_teardown_session_terminals` + the compose-down loop against *that* project's store + container
state. The existing per-store logic is correct; it just needs the outer loop.
**Warning signs:** stray `zsh`/`claude`/container processes after closing arduis with 2+ projects open.

### Pitfall 4: the global cap must count ALL projects (D-09)
**What goes wrong:** `caps.at_cap(self._store.all())` (window.py:2470) counts the active
project only; opening agents across projects silently exceeds the machine-global cap.
**Why it happens:** `active_count` takes a flat session list (caps.py:20) ‚Äî currently fed one store.
**How to avoid:** feed the **union** of every project's tasks:
`caps.at_cap([t for p in registry.all() for t in p.store.all()])`. The cap functions need no
change ‚Äî only the argument. The hibernate chooser (`_prompt_hibernate_then`) should likewise
consider all projects' active tasks (or at minimum document that it frees within the active project).
**Warning signs:** more than `ACTIVE_CAP_DEFAULT` live agents across projects with no prompt.

### Pitfall 5: removing a project with live tasks (D-10)
**What goes wrong:** dropping a project from the list without tearing down leaks its agents/containers.
**How to avoid:** D-10 ‚Äî on remove, if the project has live (ACTIVE) tasks, confirm via
`Adw.AlertDialog`; on confirm, run `_teardown_session_terminals` for each task + `compose.down_argv`
for each isolated task, then `registry.remove(root)` and rewrite `projects.json`. **Never** delete
worktrees/task dirs on disk (D-10, matches the existing "arduis never deletes" rule).
**Warning signs:** removed project's processes survive; or worse, disk dirs deleted.

### Pitfall 6: projects.json with a now-missing root
**What goes wrong:** a remembered root was deleted/moved on disk; loading it crashes or shows an
empty broken project.
**How to avoid:** D-06 ‚Äî validate each root on load (`os.path.isdir`); skip (or flag) missing
roots gracefully, and rewrite the file to drop them (or keep+grey ‚Äî Claude's discretion). Never
let one bad entry abort the whole load (mirror the tolerant `tomllib`/`json` reads already in window.py).
**Warning signs:** startup crash after a project folder is renamed.

### Pitfall 7: launch cwd auto-register ordering (D-07)
**What goes wrong:** the cwd project is double-registered or not selected as active.
**How to avoid:** D-07 ‚Äî on startup: (1) load `projects.json` ‚Üí seed registry; (2) resolve cwd
via the existing `_resolve_project` logic ‚Üí if its root is absent, auto-register it; (3) select
`last_active_project` if still valid, ELSE the cwd project, as active. Preserves today's
single-project launch as the degenerate case.

## Code Examples

### Resolve XDG config path + atomic write (reuse the proven pattern)
```python
# Source: appconfig.write_theme (src/arduis/appconfig.py:84) adapted to stdlib json
import json, os, tempfile
from gi.repository import GLib   # only in the window.py caller; projects_store can take the dir as arg

def projects_json_path() -> str:
    return os.path.join(GLib.get_user_config_dir(), "arduis", "projects.json")

def save_projects(path: str, roots: list[str], last_active: str | None) -> None:
    """Atomic temp+rename; best-effort (swallow OSError) like write_theme."""
    data = {"version": 1, "projects": [{"root": r} for r in roots],
            "last_active_project": last_active}
    try:
        d = os.path.dirname(path) or "."
        os.makedirs(d, exist_ok=True)
        fd, tmp = tempfile.mkstemp(dir=d, prefix=".arduis-projects-")
        with os.fdopen(fd, "w", encoding="utf-8") as fh:
            json.dump(data, fh, indent=2)
        os.replace(tmp, path)        # atomic
    except OSError:
        pass

def load_projects(path: str) -> tuple[list[str], str | None]:
    """Tolerant load; skip missing roots (D-06). Returns (valid_roots, last_active)."""
    try:
        with open(path, encoding="utf-8") as fh:
            data = json.load(fh)
    except (OSError, ValueError):
        return [], None
    raw = data.get("projects", []) if isinstance(data, dict) else []
    roots = [e["root"] for e in raw if isinstance(e, dict) and isinstance(e.get("root"), str)
             and os.path.isdir(e["root"])]   # skip missing (D-06)
    last = data.get("last_active_project") if isinstance(data, dict) else None
    if last not in roots:
        last = None
    return roots, last
```

## Open Questions

1. **Status-file term_id namespacing across projects (A3).**
   - What we know: `attention.state_file_path(status_dir, term_id)` builds one global path; term_ids are branch-derived and can collide across projects.
   - What's unclear: whether to namespace the status dir per project, prefix term_id with a project discriminator, or partition only the in-memory maps.
   - Recommendation: planner picks; simplest safe option is to include a project discriminator (e.g. a short hash of the root) in the term_id or the status subdir.

2. **Does the hibernate-chooser (D-09 cap gate) span projects?**
   - What we know: `_prompt_hibernate_then` lists the active project's active tasks.
   - What's unclear: whether the chooser should offer tasks from *other* projects to free the global cap.
   - Recommendation: at minimum count globally (D-09); offering cross-project hibernation in the chooser is a nicety ‚Äî defer unless trivial.

3. **`projects.json` on a missing root: drop or keep-greyed?**
   - Recommendation: skip on load (D-06) and rewrite to drop; keeping a greyed "not found" tab is a UX nicety that can defer.
