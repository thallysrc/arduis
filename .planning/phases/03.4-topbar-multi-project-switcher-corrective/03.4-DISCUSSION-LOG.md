## Core model (PO decisions, pre-locked in chat before --auto)

| Decision | Choice |
|----------|--------|
| What the topbar represents | Multiple PROJECTS (each a multi-repo root), not repos of one project |
| How projects are added | "Open project" folder picker + remembered list persisted across launches |
| Parallelism | "Both alive" — other projects keep terminals/agents/containers running; topbar switches the visible one |
| Member repos | Leave the topbar; surface only in the New-task dialog |

## Project switcher widget

| Option | Description | Selected |
|--------|-------------|----------|
| Header tab bar | One tab per project in the headerbar (reuse the chip-bar slot) + "+ Open project" | ✓ |
| Dropdown/menu | A single project menu button | |
| Left rail | A vertical project rail | |

**Auto choice:** Header tab bar — matches the mockup room-tabs affordance and reuses the existing `_chip_bar` `pack_start` slot; few projects expected.

## Project-list persistence

| Option | Description | Selected |
|--------|-------------|----------|
| JSON state file (XDG) | `~/.config/arduis/projects.json` via stdlib `json` | ✓ |
| tomli-w | Write a TOML file (adds optional dep) | |

**Auto choice:** JSON state file — keeps `.arduis.toml` read-only (CLAUDE.md `tomllib`), no new dependency.

## "Both alive" parallelism & caps

| Option | Description | Selected |
|--------|-------------|----------|
| Global cap | Existing agent/container cap spans ALL projects | ✓ |
| Per-project cap | Each project gets its own cap | |

**Auto choice:** Global cap — RAM is a machine-global resource; per-project cap deferred unless dogfooding shows the global cap is too coarse.

## Launch / backward-compat

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-register cwd | Resolve launch dir → project, add to list, select it | ✓ |
| Empty start | Start with no project; require Open | |

**Auto choice:** Auto-register cwd — preserves today's "launch lands you in it" (D-07) as the degenerate case.

## Add / remove lifecycle

| Option | Description | Selected |
|--------|-------------|----------|
| Teardown on remove (confirm if live) | Removing a project kills its agents + compose down; never deletes worktrees | ✓ |
| List-only removal | Drop from list, leave processes running | |

**Auto choice:** Teardown on remove — honors "teardown garantido"; confirm when live tasks exist.

## Project model architecture

| Option | Description | Selected |
|--------|-------------|----------|
| Per-project SessionStore behind a registry | GTK-free `Project` (root + member_repos + own store + terminals) + `ProjectRegistry` | ✓ |
| Single store keyed by project | One flat store with a project field on each task | |

**Auto choice:** Per-project store behind a registry — cleanest encapsulation, mirrors the existing GTK-free seam discipline.

## Switch UX

| Option | Description | Selected |
|--------|-------------|----------|
| Retain widgets, swap visible | Background projects' VTE widgets kept in memory (detached), instant switch-back | ✓ |
| Rebuild on switch | Destroy + rebuild the workspace each switch | |

**Auto choice:** Retain widgets — required by "both alive" + state preservation.

## Member-repo surfacing

| Option | Description | Selected |
|--------|-------------|----------|
| New-task dialog only | `detect_member_repos` feeds the dialog; topbar carries no repos | ✓ |

**Auto choice:** Locked by the GAP — repos are a task-creation concern, not a topbar concern.

## Deferred Ideas

- Per-project attention/status rollup dot on the tab.
- Command/Swarm/Review room tabs (v2).
- Drag-reorder/pin project tabs; per-project themes.
- Optional per-project cap on top of the global cap.
