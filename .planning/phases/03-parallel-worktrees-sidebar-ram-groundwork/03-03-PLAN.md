<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kernel `/proc` → resource_monitor | resource_monitor reads `/proc/<pid>/{stat,smaps_rollup,statm}`. It walks only pids whose process-group equals a `pgid` arduis itself spawned (the user's own shell/agent), never escalating privilege. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Information Disclosure | `_pids_in_group` scanning all of `/proc` | mitigate | The walk only SUMS RSS for pids whose pgrp matches the session's own `pgid`; it reads no command lines, env, or file contents. A `PermissionError` on another user's pid is swallowed (no data leaks, no crash). |
| T-03-06 | Tampering (pid reuse) | RSS read between enumerate and read | mitigate | Group membership is re-derived each call from live `/proc` `stat` pgrp (no cached pid lists); each per-pid read is wrapped in try/except (FileNotFoundError/ProcessLookupError/PermissionError → skip), so a recycled/exited pid yields 0, not a stranger's RSS or a traceback (Pitfall 3). |
| T-03-07 | Denial of Service | unbounded `/proc` scan blocking caller | accept | Work is bounded to the 5–12 process-group working set (~3 tiny virtual-file reads/pid → sub-ms, Assumption A1); the off-loop scheduling that protects the GTK main loop is window.py's concern (Plan 03-05), not this pure module. |
| T-03-08 | Elevation of Privilege | cap bypass | mitigate | `at_cap` is a pure boolean over the in-memory store; window.py (03-05) must route the +New flow through it. There is no path that reads the cap from untrusted input in Phase 3 (interim constant; Phase 6 adds trusted-repo `.arduis.toml`). |

Spoofing/Repudiation are N/A — no authentication surface; this is local single-user process accounting.
</threat_model>
