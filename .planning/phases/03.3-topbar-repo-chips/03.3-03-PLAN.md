<plan_decisions>
- **Construct the window with the project root as cwd.** `_resolve_project` reads `os.getcwd()`. The smoke
  must `os.chdir(project_root)` (in the sandbox) before constructing `ArduisWindow`, then assert on
  `win._chip_state` / `win._chip_bar`. Restore cwd in `finally`.
- **Drive interaction at the model+widget seam, not via synthetic clicks.** A real pointer click is not
  reliably drivable headlessly. Assert the chip-bar children count + the chip button `get_active()` states,
  and drive toggling by calling the SAME handler the click would (`win._on_toggle_chip`-style path or directly
  manipulating `_chip_state` + `_restyle_chips`, mirroring how test_theme_switch_smoke calls `_on_set_theme`
  and test_setup_feed_smoke calls the trust+feed path directly). Then assert `default_selection()` reflects it.
- **Three fixtures, one harness:** (a) clean multi-repo (2-3 `.git`-DIR repos) → N chips; (b) Livon-Saude
  shape (2 `.git`-DIR repos + 20 `.git`-FILE worktree subdirs) → exactly 2 chips, the 20 NOT present; (c)
  degenerate (a single `git init` root, no member subdirs) → resolves via the rev-parse fallback to a 1-repo
  project and renders its single chip without error. Build each as a sandbox subdir; `git init` only where a
  real repo is needed (the `.git`-FILE worktrees can be faked by writing a `.git` FILE — detection only checks
  isdir vs file, not validity).
- **The pointer-click render of the dialog is live-only.** Like the other smokes, the Adw.AlertDialog
  render+click cannot be driven headlessly; the smoke emulates "toggle then open dialog default" by checking
  `default_selection()` directly and (optionally) constructing the dialog's seed list the same way Plan 02
  does. Note this limitation in the harness docstring; the live checklist covers the real dialog.
</plan_decisions>
