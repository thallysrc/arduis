# GAP / Corrective: Topbar = multi-PROJECT switcher (level-1 fix)

**Raised:** 2026-06-15 by the user (PO) — second correction of the same surface.
**Corrects:** Phase 03.3 in full (toggleable repo chips) **and** the original 03.2 D-06
("topbar = project name only"). Both got the topbar's *level* wrong.
**Status:** decided — to plan & execute as a corrective phase (proposed **03.4**), BEFORE
Phase 9 (packaging ships the final v1 UX, so the topbar must be right first).

## Why

GSD has now mis-modeled the topbar twice. 03.2 made it a bare project-name string; 03.3
"corrected" that to **chips of the member repos of ONE project** (`backend`, `frontend`,
`keycloak`), wired to task-repo-selection (toggle → New-task default; active task reflected).

Both are the **wrong level**. The approved 3-level model (`docs/MOTIVATION.md`: *"Topbar =
projetos. Um projeto é uma pasta raiz multi-repo"*) means the topbar lists **PROJECTS**, where
each project is itself a whole multi-repo. The repos *inside* a project are not a topbar concern —
they surface only when **creating a task**.

PO's words: *"a topbar do meu app não é a lista de repositório de uma task, é um multi
repositório… na topbar existiria Livon-Saude (o multirepo), e nas tasks (sidebar) teria os clones
desse Livon-Saude. Pois eu posso mexer na Livon e na KarveLabs, dois projetos diferentes rodando
em paralelo."*

```
Topbar:    [ Livon-Saude ]  [ KarveLabs ]  …    ← each item = one multi-repo PROJECT
Sidebar:     tasks of the selected project       ← worktrees/clones of that multi-repo
Workspace:   terminals of the selected task
```

## Decided design (PO answers, 2026-06-15)

- **Topbar = a switchable list of projects** (multi-repo roots), not repos of one project.
  Selecting a project swaps the sidebar (its tasks) + workspace.
- **Adding a project = "Open project" folder picker + a remembered list** persisted across
  launches. One arduis instance covers MANY projects (kills the "one arduis = one launch-dir
  project" premise).
- **"Both alive" parallelism:** Livon and KarveLabs keep their terminals/agents/containers
  **running simultaneously**; the topbar only chooses which is *visible*. State is preserved
  across switches (deliberate RAM tradeoff — see below). Per-task hibernation still applies.
- **Member repos leave the topbar.** `backend`/`frontend`/`keycloak` appear only in the
  "Nova task" dialog (where the user picks which repos the task spans). No more topbar chips.

## Architectural implications (this is NOT a topbar tweak)

The "one arduis = one project" premise is baked as singletons in `window.py`:
`_project_root` (one str), `_member_repos` (one list), `_repo_root`, and `_resolve_project()`
runs ONCE at startup from `cwd`; the task `_store` is single-project.

1. **Introduce a real `Project` model** — each project owns its own `root`, `member_repos`,
   task store, and live terminals. The topbar switches the active `Project`; the window swaps
   sidebar + workspace while keeping the other projects' terminals alive in the background.
   Today none of this is encapsulated. (Keep it GTK-free like `SessionStore`/`topbar.py`.)
2. **Project-list persistence (NEW)** — "open + remember" must WRITE state. CLAUDE.md pins
   `tomllib` (read-only). Writing the remembered-projects list needs either `tomli-w` (already
   flagged optional in the stack) or a separate state file. Decide in planning; note disk is the
   source of truth elsewhere (no app-state file convention exists yet — see `_scan_tasks`).
3. **RAM-first tension** — "both alive" means switching projects does NOT tear down the other's
   agents/containers. PO chose this knowingly (state preserved). Mitigate with the existing
   configurable active caps + per-task hibernate; consider surfacing aggregate RAM per project.

## What survives from 03.3 / what dies

- **SURVIVES:** `project.detect_member_repos` + the D-04 `.git`-DIRECTORY filter (still needed to
  know a project's repos at task-creation time). The status-dot CSS classes.
- **DIES / repurposed:** the topbar *render* half becomes a **project bar** (one item per
  project, switchable) instead of repo chips. `topbar.py ChipState` (toggle→seed
  `default_selection`, `reflect_active` task coupling) largely goes away — its job was the wrong
  level. The New-task dialog goes back to picking repos itself (no topbar seeding).
