<plan_decisions>
- **Chip widget shape:** each chip is a `Gtk.ToggleButton` whose child is a small `Gtk.Box` of a dot `Gtk.Label("●")` (reusing `arduis-dot-*`) + a name `Gtk.Label` (`set_text`, never `set_markup`). The ToggleButton's `active` state mirrors `ChipState.is_selected`; its `toggled` handler calls `self._chip_state.toggle(repo)` then restyles. Reflection (D-03) adds/removes a CSS class (e.g. `arduis-chip-active`) by `is_active`. A ToggleButton gives the on/off affordance for free and is keyboard-accessible.
- **Where chips live:** pack the chip bar as a single horizontal `Gtk.Box` (named `self._chip_bar`) at `pack_start` AFTER `self._new_btn`, and KEEP the project name as the `Adw.WindowTitle` (centered). This leaves `pack_end` (and the title-adjacent region) free for 07-04's container toggle + port badges. Do NOT replace the WindowTitle with the chips — the name "stays visible" (D-01) and the centered title is the natural co-existence anchor with badges.
- **Status bolinha state for now:** the chip dot reflects the same per-task status the sidebar shows where a task for that repo is active; with no active task it is the neutral `arduis-dot-active`/`arduis-dot-hibernated` (active green when the project is resolved). Full per-repo container/agent status badges are Phase 7 / Phase 4 territory — do NOT build new status plumbing here; reuse the existing dot classes as a static-but-correct affordance. (No "v1 placeholder" language — this fully delivers D-01's "status bolinha reusing the sidebar dot classes"; richer per-repo status is a separate, already-scoped phase.)
- **Overflow (D-05):** measure nothing fancy — keep a configurable cap (e.g. show up to N chips, default 6 to match the mockup) and, when `len(members) > N`, render the first N chips + a `Gtk.MenuButton` labelled `+{len-N}` whose `Gio.Menu` lists the remaining repos as toggle items (`win.toggle_chip(repo_name)` action, mirroring `win.set_theme`). Horizontal scroll is explicitly rejected (CONTEXT D-05 — fiddly with VTE focus). The cap is a module constant so it is trivially tunable.
- **Rebuild on resolution + on reflection:** build the chip bar once `_member_repos` is known (end of `_resolve_project`), and have a `_restyle_chips()` that re-applies selected/active CSS in place (no full rebuild) on toggle and on workspace swap. A full `_rebuild_chip_bar()` is only needed if the member set changes (it does not during a session — one arduis = one project).
</plan_decisions>
