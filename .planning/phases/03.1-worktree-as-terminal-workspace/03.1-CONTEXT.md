# Phase 03.1: worktree-as-terminal-workspace - Context

**Gathered:** 2026-06-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Pivot the workspace model to match the approved 3-level interface in `docs/MOTIVATION.md`:
**the pane canvas shows the terminals of ONE worktree at a time** (workspace), not multiple
worktrees side by side. Selecting a worktree in the sidebar swaps the entire workspace.
Each worktree owns its own set of terminals (default 2: agent + shell) and its own layout tree.

This **supersedes the Phase-03 semantics** where each pane was a different worktree
(D-02/D-03/D-06 of Phase 03 are re-targeted: visibility decoupling, split-on-create and
focus-or-swap now operate at the new level — see decisions below).

**In scope:** levels 2-3 of the model (sidebar = worktrees, workspace = selected worktree's
terminals). **Out of scope:** level 1 (topbar = multiple repos) — deferred to its own phase;
the repo continues to be resolved from arduis's launch cwd (Phase-2 D-03).

</domain>

<decisions>
## Implementation Decisions

### Scope
- **D-01:** Topbar/multi-repo (level 1) is **OUT** of this phase. Only the worktree→workspace
  pivot ships. Repo resolution stays cwd-based (Phase-2 D-03).

### Default workspace composition
- **D-02:** A new worktree's workspace starts with **2 terminals, side by side (horizontal
  split)**: **left = zsh with `claude` fed in** (existing spawn path, Phase-2 D-08),
  **right = plain zsh** in the worktree dir. Mirrors the user's backend/frontend tmux habit.
- **D-03:** The 2nd terminal **spawns eagerly** (born together with the workspace). A plain
  zsh is ~10-20 MB; the real RAM cost is the agent. No lazy-spawn states.

### Per-worktree layout
- **D-04:** **Each worktree owns its own layout tree** (reuse the GTK-free `layout.py` binary
  tree — one instance per worktree instead of one global). Switching workspaces away and back
  **preserves splits, zoom, and pane focus** — like tmux windows.
- **D-05:** **New splits inside a workspace spawn zsh + `claude` fed in** (same as the primary
  terminal). Every split is an agent terminal by default; Ctrl+C drops to the shell as always.

### Keybindings (re-targeted, same C-Space prefix mechanism from Phase 3)
- **D-06:** `C-Space h/j/k/l` moves focus **between the terminals of the visible worktree**.
  `C-Space n/p` and `C-Space <number>` switch **the whole workspace (worktree)**. tmux mental
  model: panes = terminals, windows = worktrees. Prefix state machine and keymap constants
  from Phase 3 are reused; only action targets change.

### Pinned `main` row
- **D-07:** The pinned `main · zsh` row becomes a **regular workspace** with 1 terminal (the
  $HOME scratch zsh, no claude). Selecting it swaps the workspace like any worktree; it can be
  split manually like any workspace. No special-case UI code.

### Hibernate / resume / RAM (re-targeted from Phase-2 D-10..D-12, Phase-3 D-12..D-16)
- **D-08:** **Hibernate kills ALL the worktree's terminals** (every process group: agent +
  shells, SIGHUP→SIGKILL grace), keeping the directory on disk. Hibernated = zero RAM cost.
- **D-09:** **Resume relaunches the default layout** (claude + shell, D-02). Extra splits are
  not restored — re-create manually. (Layout memory D-04 applies only while active; hibernate
  discards the saved tree.)
- **D-10:** The sidebar row's RAM figure is the **sum of all the worktree's terminal process
  groups**; the footer aggregate sums across worktrees as before.
- **D-11:** The active-agents **cap counts active (non-hibernated) WORKTREES**, not terminals.
  Splits within a worktree never hit the cap; their cost is visible in the row's RAM number.
  Cap default/storage unchanged (Phase-3 D-15/D-16).

### Claude's Discretion
- How the per-worktree layout trees are stored/keyed on `SessionStore`/sessions, and how the
  GTK widget swap is implemented (e.g. `Gtk.Stack` of per-worktree canvases vs rebuild on
  switch) — must not leak or orphan VTE widgets/processes.
- Whether multiple terminals per worktree are modeled as multiple pgids on `WorktreeSession`
  or a child "terminal" record — must keep the store GTK-free and serializable.
- What `C-Space <number>` jumps to exactly (sidebar row order, as in Phase 3) and zoom/preset
  behavior within a workspace.
- Min pane sizes, divider behavior, and visual details within the Dracula palette.
- Migration mechanics from the current global layout tree to per-worktree trees.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/arduis/layout.py` (227 lines, GTK-free binary split tree) — reuse as-is, instantiated
  **per worktree** instead of one global tree; leaves become terminals instead of worktrees.
- `src/arduis/session.py` (`SessionStore`, `WorktreeSession`) — extend to model N terminals
  (pgids) per worktree; stays GTK-free/serializable.
- `src/arduis/keymap.py` + the prefix state machine in `window.py` (`_on_key`, `_run_action`)
  — same mechanism, re-targeted actions (D-06).
- `src/arduis/resource_monitor.py` — already walks process groups; sum N pgids per worktree.
- Spawn path (`spawn.py`, `_spawn_into`, `AGENT_FEED`) — unchanged; used for both default
  terminals and new splits.

### Established Patterns
- Logic lives in GTK-free modules; `window.py` widgets are a thin reflection of the model
  (`_reflect_layout`/`_build_widget`). The pivot should preserve this: per-worktree trees in
  the model, widget layer swaps the visible canvas.
- Hibernate teardown: SIGHUP→SIGKILL on pgid (`_teardown_pgid`) — extend to iterate all of a
  worktree's pgids.

### Integration Points
- `window.py` (1246 lines) concentrates the impact: sidebar `_on_row_activated` (becomes
  workspace switch), `_open_and_add` (creates 2 terminals + per-worktree tree),
  `_focus_or_swap_session` (obsolete in its Phase-3 form), `_apply_preset`/zoom (now within a
  workspace), `_poll_ram`/`_update_footer` (sum per worktree), `_on_hibernate`/`_on_resume`
  (all pgids / default layout).

</code_context>

<specifics>
## Specific Ideas

- tmux mental model is the anchor: **panes = terminals of a worktree, windows = worktrees**
  (user is tmux-centric; `C-Space` prefix, `h/j/k/l`).
- "claude + shell side by side" mirrors the user's observed habit (backend/frontend panes).
- Every new split is an agent terminal (D-05) — the user wants agents at zero friction, and
  Ctrl+C→shell already covers the plain-shell case.

</specifics>
