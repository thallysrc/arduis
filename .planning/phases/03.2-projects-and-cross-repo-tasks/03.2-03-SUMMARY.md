# Phase 03.2 Plan 03: Full Multi-Repo Task Lifecycle Summary

**Re-targeted the whole task lifecycle in `window.py` so the unit of hibernate/RAM/cap/teardown is the TASK across all its repos — per-task hibernate kills every group (no orphan), resume rebuilds the default workspace via the shared `_build_task_workspace`, RAM/cap count tasks (not repos), startup rediscovers `../<root>-tasks/` as HIBERNATED without spawning, and close-a-repository kills only that repo's groups while never deleting from disk — capped by a live + headless acceptance pass that proved no orphans with real zsh process groups.**

## Performance

- **Duration:** ~30 min (Tasks 1-2 + checkpoint-feedback iterations + final headless verification)
- **Tasks:** 3 (Tasks 1-2 autonomous; Task 3 manual GTK acceptance — partially live-verified by the user, remainder verified headless)
- **Files modified:** 1 (`src/arduis/window.py`)

## Accomplishments
- **Per-task hibernate (`_on_hibernate`):** `_teardown_session_terminals` SIGHUPs every terminal group across the task (`_all_task_terminals`), `hibernate_fields` flips state + clears every pid/pgid, the saved `LayoutModel` is discarded (`_layouts.pop`), every `{task}:` widget is dropped, and the canvas falls back to the pinned main workspace if the hibernated task was visible.
- **Per-task resume (`_resume_task`):** rebuilds the DEFAULT workspace through the shared `_build_task_workspace` (fresh `default_task_terminals` + fresh `LayoutModel` of the agent-over-shell pair) and re-spawns eagerly — only the agent is fed `AGENT_FEED`. The repo set is intact (hibernate only cleared pid/pgid). Extra splits are not restored (03.1 D-09).
- **Per-task RAM (`_poll_ram`/`_update_footer`):** each active task's RAM is the SUM of every terminal's process group across the task-level pair + any per-repo splits; the footer counts active TASKS via `caps.active_count` (a fat multi-repo task counts as 1 — D-14).
- **Cap-over-tasks (RAM-02/D-14):** `_on_new_worktree_clicked` and `_resume_gated` both route through `_prompt_hibernate_then` when at cap (the resume menu path previously skipped the cap — fixed).
- **Startup scan (`_scan_tasks`/`_dir_is_task`, D-11):** rediscovers `../<root>-tasks/` at launch; a valid task dir holds ≥1 child whose `.git` is a FILE (real worktree pointer — symlink-only dirs rejected, T-03.2-13); rediscovered tasks are HIBERNATED with NO pid/pgid (never spawns — Pitfall 6). Sidebar rebuilt after the scan so they appear immediately.
- **Close-a-repository (`_on_close_repo`, D-10):** a per-repo submenu entry kills only that repo's terminal groups (SIGHUP→SIGKILL), drops their leaves, clears their pid/pgid; the `RepoCheckout` and its dir stay. No filesystem-removal call anywhere.
- **No-orphan teardown on window close (`_on_close_request`):** stops the RAM poll, tears down the scratch shell, then every task's every terminal group via `_teardown_session_terminals`.

## Deviations from Plan

All four groups below are **checkpoint-feedback deviations** — discovered during the user's live GTK acceptance run (Task 3) and decided by the user on the spot. They supersede or correct earlier-plan behavior.

### 1. [Checkpoint feedback - Bug] Sidebar invisible to rediscovered tasks until next rebuild
- **Found during:** Task 3 live UAT (startup scan, D-11)
- **Issue:** `_scan_tasks` added rediscovered tasks to the store, but the sidebar had already been built from the (empty) store earlier in `__init__`, so scanned tasks did not appear until the next unrelated `_rebuild_sidebar`.
- **Fix:** `_scan_tasks` now calls `self._rebuild_sidebar()` after populating the store.
- **Files modified:** `src/arduis/window.py`
- **Committed in:** `abbce26`

### 2. [Checkpoint feedback - Bug + Missing Critical] Ref-namespace conflict + create-failure cleanup
- **Found during:** Task 3 live UAT (task creation across repos)
- **Issue:** A git ref-namespace D/F conflict (e.g. `feat` exists where `feat/x` is requested) produced an unclear failure; a failed create could leave a zombie task row counting toward the cap and a dead empty column in the workspace.
- **Fix:** `resolve_repo_add` now detects the D/F conflict and aborts with a clear pt-BR message; create-failure handling prunes the failed column and removes the zombie task from the store (never deleting anything from disk — D-10).
- **Files modified:** `src/arduis/window.py` (+ `src/arduis/task_layout.py` resolve logic)
- **Committed in:** `c3edc9a` (detect conflict) + `bd19b1d` (prune zombie task / failed column)

### 3. [Checkpoint feedback - Decision] Explicit resume — browsing never spawns + resume cap-gated
- **Found during:** Task 3 live UAT (sidebar navigation to a hibernated task)
- **Issue:** Clicking a hibernated task row would swap to a blank workspace (no layout/terminals); separately, the resume MENU path skipped the active-agent cap.
- **Fix (user decision):** a sidebar click on a HIBERNATED task NAVIGATES to an explicit "Retomar task" card (`_show_hibernated_placeholder`) — browsing never spawns; deliberate resume goes through the shared `_resume_gated` so it is cap-gated identically to +New.
- **Files modified:** `src/arduis/window.py`
- **Committed in:** `e7d0b39` (cap-gated resume) + `f572559` (explicit Retomar card)

### 4. [Checkpoint feedback - Architectural / UX PIVOT, supersedes D-01/D-02] One 2-terminal pair at the task root
- **Found during:** Task 3 live UAT (a real 6-repo project)
- **Issue:** The planned one-column-per-repo 2×N default produced an unusable 2×6 grid of tiny panes on a real 6-repo project.
- **Fix (user decision):** the task workspace now opens with ONE 2-terminal pair (claude over zsh) at the TASK FOLDER ROOT regardless of repo count; the user adds terminals via splits. RAM/teardown/close-repo iterate `_all_task_terminals` (task-level pair + any per-repo split). A `RepoCheckout` is now worktree METADATA only.
- **Files modified:** `src/arduis/window.py` (+ `src/arduis/session.py` `default_task_terminals`, `Task.terminals`)
- **Committed in:** `a67cd63` (task-level terminals) + `a3594eb` (workspace opens with the single pair at root)
