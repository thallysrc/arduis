# GAP / Corrective: Topbar repo chips (design-alignment)

**Raised:** 2026-06-14 by the user (PO) during manual testing of the multi-repo flow.
**Corrects:** Phase 03.2 decision D-06 ("topbar = project name only, no switcher").
**Status:** decided — to plan & execute as a corrective phase (proposed **03.3**), AFTER Phase 7's
domain (07-01/02/03) lands and BEFORE 07-04 (both touch window.py → must be sequential; doing the
topbar first avoids 07-04 building its container UI on a topbar that's about to change).

## Why

The approved design (`docs/mockup/interface-v2-bridgespace.png` + `docs/MOTIVATION.md` 3-level
model) surfaces repo/project selection in the TOPBAR as visible chips/dots. GSD's 03.2
discuss-phase flattened the topbar to a bare project-name string (D-06), so the multi-repo nature
is invisible until the "Nova task" dialog opens. The user flagged the divergence. MOTIVATION is
also internally inconsistent (revised "Topbar = projetos" vs leftover line 143 "Repos no topbar").

## Decided design (PO choice: "Chips de repo no topbar, como o mockup")

- Topbar shows **one chip per project member repo** (e.g. `backend`, `frontend`, `IAM`), each with
  a status bolinha, **toggleable**.
- The toggled-ON set is the **default repo selection** for the next "Nova task" — the dialog still
  exists and inherits the toggled set (per-task override preserved). "Select multi-repo in the
  topbar" without losing per-task flexibility.
- When a task is selected, its repos are reflected/highlighted in the chips.
- Container badges from Phase 7 (07-04: `db :5433`, `compose shared`) sit ALONGSIDE the repo chips
  on this same surface (the mockup puts them together) — so 07-04's UI half should target the new
  topbar, hence the sequencing.

## Mandatory sub-fix: repo detection must filter linked worktrees

`detect_member_repos` currently counts EVERY direct subdir with a `.git` (dir OR file). The user's
real `Livon-Saude` root has ~20 linked worktrees (`backend-*`, `frontend-*` — `.git` is a FILE),
which would flood the topbar with 20 chips. **Decision:** member-repo detection counts only subdirs
whose `.git` is a **directory** (a true repo), excluding `.git`-file worktrees. (This was the
earlier "Filtrar worktrees" option, now mandatory for chips to be usable.) Verify it doesn't break
the degenerate 1-repo case or the existing 03.2 tests.
