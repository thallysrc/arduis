# Phase 03.2: Projects and Cross-Repo Tasks - Research

**Researched:** 2026-06-11
**Domain:** Multi-repo project model + cross-repo git worktree "tasks" on top of an existing GTK4/PyGObject/VTE desktop app
**Confidence:** HIGH (this is an in-codebase refactor/extension; the unknowns are git-worktree mechanics + symlink layout, both verifiable locally)

## Summary

Phase 03.2 is **not greenfield** — it is a structural refactor of an already-working Phase 03.1
app. The unit of work pivots from `WorktreeSession` (one branch → one worktree dir → N terminals)
to a **Task** (one branch name → N member repos → N worktree dirs → 2 terminals each), and the
"repo" concept becomes a **Project** = a root folder whose direct subdirs containing `.git` are
member repos. The existing GTK-free modules (`session.py`, `worktree.py`, `caps.py`,
`resource_monitor.py`, `spawn.py`, `host_runner.py`) were explicitly designed to be extended this
way; `window.py` is the only place where real new orchestration lands (per-repo loops + symlink
materialization + project-root resolution + task-scan-at-startup).

The two genuinely new technical surfaces are (1) **`git worktree add` invoked per member repo into
a shared task folder** — each repo keeps its `.git` pointing back to its original repo, so the
worktrees are independent and `git worktree list` in each member tracks them separately; and (2)
the **task-folder layout** that mirrors the project root via symlinks (top-level files, non-repo
dirs, and unchosen member repos symlinked to the original; chosen repos are real worktrees) so root
tooling (`CLAUDE.md`, `docker-compose.yml`, relative build contexts) resolves verbatim inside the
task. Both are shell-out + filesystem operations the codebase already has idiomatic seams for
(`run_git_async` through `HostRunner`, GTK-free argv builders in `worktree.py`).

**Primary recommendation:** Model `Task` and `RepoCheckout` as new GTK-free dataclasses in
`session.py` alongside (not replacing) `WorktreeSession`; add a `task_worktree_dir_for(...)` builder
+ a pure symlink-plan function to `worktree.py`; drive a **per-repo async chain** in `window.py`
(reusing the existing `_resolve_base_then_add` → `_open_and_add` machinery repo-by-repo); keep the
1-repo path identical to 03.1 by making the single-repo project a Task with exactly one
`RepoCheckout` (no special-case branch). Use `os.path.relpath`-based **relative symlinks** so the
whole task folder is relocatable and survives the user moving it.

## Architecture Patterns

### Recommended Model Structure (GTK-free, in `session.py`)
```
Project (the cwd root; topbar shows project_name)
  └── member_repos: list[str]   # dir names of direct subdirs with .git (D-05)
Task (the new unit; sidebar row; RAM/cap/hibernate target)
  ├── task_id / branch          # one branch name across all chosen repos
  ├── task_dir                  # ../<root>-tasks/<sanitized-branch>/
  ├── state: ACTIVE | HIBERNATED
  └── repos: list[RepoCheckout]
        RepoCheckout
          ├── repo_name         # e.g. "backend"
          ├── worktree_dir      # <task_dir>/<repo_name>
          ├── branch
          └── terminals: list[TerminalRecord]   # [agent t0, shell t1] (D-01)
```
The **1-repo project is a Task with one RepoCheckout** — no special-case branch (D-15 of CONTEXT
success criterion 5). Terminal ids generalize from `{branch}:tN` to `{branch}:{repo}:tN` (or keep a
flat `{task}:tN` index and store the owning repo on the `TerminalRecord` — see Open Question 1).

### Pattern 1: Project-root resolution (replaces `_resolve_repo_root`, window.py:863)
**What:** At startup, scan `os.getcwd()`'s direct subdirs for `.git`; that set is the member repos.
**When:** Once at startup, blocking is acceptable (CLAUDE.md — same as today's single `git rev-parse`).
**Rule (D-05/D-07):** No tree-walk-up. The root's own `.git` is NOT a member. If zero subdir members
but cwd itself is a git repo → degenerate 1-repo project (preserve 03.1).
```python
# GTK-free helper (new), called from window.py at startup
def detect_member_repos(root: str) -> list[str]:
    members = []
    with os.scandir(root) as it:          # direct subdirs only (D-05/D-07: no walk)
        for e in it:
            if e.is_dir() and os.path.isdir(os.path.join(e.path, ".git")):
                members.append(e.name)
    return sorted(members)
# If members == [] and root itself is a git repo → degenerate single-repo task.
```
[ASSUMED — pattern derived from D-05/D-07; not a library API. Verify `.git` may be a *file* in a
submodule/worktree, so test `os.path.exists(.../.git)` not only `isdir` — see Pitfall 4.]

### Pattern 2: Per-repo task creation as a CHAINED async sequence
**What:** Reuse the existing `_create_worktree → _continue_create_worktree → _resolve_base_then_add →
_open_and_add` flow, but loop it **repo by repo** (D-13 resolves new-vs-existing/checked-out *per
repo*). Each step is a `run_git_async` whose callback launches the next repo — never threads
(CLAUDE.md).
**When:** "New task" confirmed with a branch + a set of chosen repos.
**Order:** materialize the task folder + symlinks FIRST (so relative compose/build contexts resolve),
then per-repo `worktree add`, then spawn that repo's 2 terminals.
```python
# Sketch — chained continuation, one repo at a time (no asyncio/threads)
def _add_repo(self, task, repos, i, ...):
    if i >= len(repos):
        return self._finalize_task(task)   # all repos checked out + spawned
    repo = repos[i]
    # per-repo D-13: porcelain pre-check → new-vs-existing → base → add
    argv = argv_worktree_add_new(...) or argv_worktree_add_existing(...)
    def _done(status, out, err):
        if status != 0:
            self._abort_repo(repo, err)    # clear message, never --force (D-13)
        else:
            self._spawn_repo_terminals(task, repo)
        self._add_repo(task, repos, i + 1, ...)   # next repo
    run_git_async(argv, _done, self._runner)
```

### Pattern 3: Task-folder materialization via relative symlinks (D-08/D-09)
**What:** Create `<parent>/<root_base>-tasks/<sanitized-branch>/`; for every top-level entry of the
project root that is NOT a chosen member repo, create a symlink pointing at the root original. Chosen
repos become real worktrees inside the same folder under their dir name.
**Why relative symlinks:** the whole task folder is relocatable (the user moves/cleans up manually,
D-10) and survives the project root being on the same volume. Use `os.symlink(os.path.relpath(src,
start=task_dir), dst)`.
```python
def symlink_plan(root: str, task_dir: str, chosen_repos: set[str]) -> list[tuple[str, str]]:
    """(src_abs, dst_in_task_dir) for every root entry except chosen repos.
    Chosen repos are materialized as worktrees, not symlinks (D-09)."""
    plan = []
    for name in os.listdir(root):
        if name in chosen_repos:
            continue                       # real worktree, handled separately
        if name == ".git":
            continue                       # meta-repo .git is NOT mirrored (D-05)
        plan.append((os.path.join(root, name), os.path.join(task_dir, name)))
    return plan
# materialize: os.symlink(os.path.relpath(src, task_dir), dst)
```
[ASSUMED — symlink-relative recommendation; verify on the user's real `livon/` layout that compose
relative `build:` contexts and bind mounts resolve through the symlinks before locking.]

### Pattern 4: `git worktree add` targets a path INSIDE the task folder
Each chosen repo's worktree dir is `<task_dir>/<repo_name>` (D-08). `git worktree add` records the
new worktree against the **original** member repo's `.git` (the worktree's `.git` is a *file*
pointing back into the member repo's `.git/worktrees/...`). Consequence: `git worktree list` run in
the original member repo lists the task's worktree; teardown/cleanup is still a `git worktree`
concern in the original repo, which is why D-10 keeps deletion manual for now.
[VERIFIED: `git worktree --help` — "the new worktree is associated with a branch named after
$(basename <path>)" and worktrees refuse to check out a branch checked out elsewhere unless
`--force`; we pass an explicit branch and never `--force` (D-13).]

### Anti-Patterns to Avoid
- **Special-casing the 1-repo project.** Success criterion 5 demands the single-repo case be a Task
  with one RepoCheckout, reusing the identical code path. Do not branch on `len(repos) == 1`.
- **Parallel per-repo git via threads/asyncio.** CLAUDE.md forbids it; chain `run_git_async`
  callbacks instead. (Parallelism would also make D-13's per-repo error reporting racy.)
- **Absolute symlinks + assuming the task folder never moves.** D-10 says the user moves/deletes
  task folders by hand; relative symlinks survive that.
- **Mirroring the meta-repo's own `.git`** into the task folder — D-05 excludes it; symlinking it
  would make the task folder look like the root repo and confuse git.
- **Hand-rolling worktree-dir/branch sanitization** — `sanitize_branch_for_dir` already exists and is
  tested (path-traversal guard T-02-02). Reuse it for the task-dir leaf.

## Common Pitfalls

### Pitfall 1: `.git` is a FILE in a worktree (and sometimes a submodule), not always a dir
**What goes wrong:** `detect_member_repos` using `os.path.isdir(.../.git)` misses repos whose `.git`
is a file (linked worktrees, submodules). It also could wrongly treat the project root as a member.
**Why:** git worktrees store `.git` as a pointer *file*; submodules too.
**How to avoid:** Test `os.path.exists(os.path.join(subdir, ".git"))` (file OR dir). For the
degenerate-root check, run `git -C <root> rev-parse --is-inside-work-tree` (the existing
`rev-parse` idiom). [VERIFIED: git worktree layout — worktree `.git` is a file with `gitdir:`.]
**Warning signs:** A real repo subdir not appearing in the multi-pick dialog.

### Pitfall 2: A branch checked out in ANOTHER worktree of the SAME repo blocks `worktree add`
**What goes wrong:** Creating task `feat-x` when `feat-x` is already checked out in `backend` (e.g. a
prior task) → `git worktree add` refuses (exit non-zero) unless `--force`.
**Why:** git forbids the same branch in two worktrees of one repo.
**How to avoid:** D-13 already mandates this: porcelain pre-check per repo via
`argv_worktree_list_porcelain` + `branch_checked_out_path`; if checked out elsewhere, **abort that
repo with a clear message, never `--force`.** This is partial-success: some repos may succeed and
one aborts — decide whether to roll back or leave the partial task (see Open Question 2).
**Warning signs:** Cryptic `fatal: '<branch>' is already checked out at ...`.

### Pitfall 3: Partial task creation (one repo's `worktree add` fails mid-chain)
**What goes wrong:** Repo 1 succeeds + spawns terminals, repo 2 fails. The task is half-materialized.
**Why:** Per-repo chain has independent failure points (born-HEAD, checked-out-elsewhere, disk).
**How to avoid:** Decide the policy up front. Options: (a) **best-effort** — keep the repos that
succeeded, show which repo aborted and why (matches D-10's "arduis never deletes"; the user fixes the
failed repo manually). (b) abort-all (would require deletion — contradicts D-10). Recommend (a):
materialize what works, surface a clear per-repo error, never auto-rollback. [ASSUMED — needs
confirmation; D-10's no-delete stance strongly implies best-effort.]
**Warning signs:** A task row whose RAM/terminal count is lower than the chosen-repo count.

### Pitfall 4: Symlinked compose/build-context paths must resolve relative to the task folder
**What goes wrong:** Root `docker-compose.yml` symlinked into the task folder has `build: ./backend`
or `volumes: ./scripts:/x`. If `backend` is a real worktree dir but `scripts` is a symlink, paths
must still resolve. If symlinks are absolute and the folder is moved, they break.
**Why:** Compose resolves relative paths against the compose file's *directory* (the task folder).
**How to avoid:** Mirror the *entire* root layout (every top-level entry symlinked except chosen
repos, per D-09) so every relative path the compose/CLAUDE tooling uses exists at the same relative
position. Use **relative** symlinks. Phase 7 consumes this; 03.2 only has to keep paths resolving.
[CITED: docs.docker.com/compose — relative paths resolve against the compose file location; combined
with D-09's "root tooling runs verbatim".]
**Warning signs:** `docker compose config` (Phase 7) erroring on missing build context inside a task.

### Pitfall 5: VTE widget id collisions when terminal ids gain a repo dimension
**What goes wrong:** Today ids are `{branch}:tN` and the teardown/hibernate code does prefix matching
on `f"{sid}:"` (window.py:1438). If a task spans repos and ids become `{branch}:{repo}:tN`, the
prefix-match and `_next_term_id` logic must account for the repo segment or risk collision/leak.
**Why:** The map cleanup in `_on_hibernate` iterates `t.startswith(prefix)`.
**How to avoid:** Pick ONE id scheme and update `_next_term_id`, the hibernate prefix sweep, and
`default_terminals` consistently. Recommend `{task}:{repo}:tN` and store `repo_name` on
`TerminalRecord` so RAM/teardown can group by repo without string parsing.
**Warning signs:** Hibernating one task tears down another's terminals, or a VTE widget orphan after
hibernate.

### Pitfall 6: Startup task-scan must not spawn terminals (tasks start HIBERNATED)
**What goes wrong:** Eagerly spawning every discovered task's terminals at startup blows the RAM
budget and ignores D-11 (tasks listed as HIBERNATED, resume relaunches).
**Why:** D-11 + D-14 — disk has every past task; only resume spawns.
**How to avoid:** Scan → create `Task` objects in `HIBERNATED` state with empty/`pgid=None`
terminals → render dimmed rows. Resume (existing `_on_resume` path, re-targeted) spawns the default
layout for ALL the task's repos.
**Warning signs:** App startup spawns dozens of `claude` processes.

```python
import os

def detect_member_repos(root: str) -> list[str]:
    members = []
    try:
        with os.scandir(root) as it:
            for e in it:
                if e.is_dir(follow_symlinks=False) \
                   and os.path.exists(os.path.join(e.path, ".git")):  # file OR dir
                    members.append(e.name)
    except OSError:
        return []
    return sorted(members)
# Caller (window.py startup): if not members and cwd is itself a repo → 1-repo task.
```

|---------|--------|---------------------|

| Malicious dir name in scanned subdirs (e.g. with newlines) used in argv | Tampering | git argv pass dir names as discrete `-C`/path elements; never interpolate into a string |
