# Phase 03.2: Projects and Cross-Repo Tasks - Context

**Gathered:** 2026-06-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Pivot level 1 of the interface (docs/MOTIVATION.md, revised 2026-06-10): **project = multi-repo
root folder** (e.g. a root holding `backend/`, `frontend/`, `keycloak/` + root `CLAUDE.md` /
`docker-compose.yml`, ideally a small meta-repo). Introduce **task** as the unit of work:
creating a task picks 1+ member repos, creates same-named branches/worktrees in each, and
materializes a **task folder mirroring the root layout**. Sidebar pivots from worktrees to
**tasks of the current project**; the workspace shows the selected task's terminals.
Hibernate/RAM/cap/teardown re-target from worktree to task. Single-repo project is the
degenerate case — Phase 03.1 behavior preserved with zero special-casing.

**Out of scope:** project switcher / multiple simultaneous projects (topbar shows the current
project's name only); task deletion from the UI (user cleans up manually); compose
orchestration (Phase 7); attention status (Phase 4).

</domain>

<decisions>
## Implementation Decisions

### Default task terminals
- **D-01:** A multi-repo task opens with **agent + shell per chosen repo** (2 repos = 4
  terminals: claude+zsh of backend, claude+zsh of frontend). Mirrors the user's real tmux
  habit (multiple claudes in parallel). 1-repo task = exactly the 03.1 default.
- **D-02:** Default layout: **one column per repo** (agent on top, shell below); 1-repo task
  stays side-by-side as in 03.1 D-02. This is a **default, not a rule** — "repo = column" is
  not structural; terminals are freely splittable/rearrangeable afterwards.
- **D-03:** **All agents spawn eagerly** (consistent with 03.1 D-03 — no lazy states). The
  cost is visible in the task row's RAM figure; the active-task cap bounds the total.
- **D-04:** A new split **inherits the cwd of the focused terminal's repo** (tmux-like
  locality). Splits still spawn zsh + `claude` fed (03.1 D-05).

### Project & member-repo discovery
- **D-05:** Members are **auto-detected**: every direct subdirectory of the root containing
  `.git` is a member repo. The root's own `.git` (meta-repo) is NOT a member. `.arduis.toml`
  may refine this later (Phase 6).
- **D-06:** Project opening stays **cwd-based**: launching arduis in the root folder defines
  the session's project. The **topbar is born structural** — it displays the project name —
  but has **no switcher**; multiple/recent projects are a future phase.
- **D-07:** **cwd = root, no magic.** arduis never walks up the tree. A cwd that is itself a
  git repo (no member subrepos) = degenerate 1-repo project (03.1 behavior).

### Task folder materialization
- **D-08:** Task folders live in a **grouped sibling dir: `../<root>-tasks/<branch>/`**, with
  the worktrees INSIDE the task folder using the repos' dir names (e.g.
  `livon-tasks/feat-x/backend`, `livon-tasks/feat-x/frontend`). Evolves the Phase-2
  `../<repo>-<branch>` sibling pattern without polluting the parent dir.
- **D-09:** **Everything else is symlinked to the root original**: top-level files
  (`CLAUDE.md`, `docker-compose.yml`, `.env`…), non-repo dirs (`scripts/`, `docs/`…) AND
  member repos NOT chosen for the task. One rule; root tooling runs verbatim inside the task;
  root changes reflect immediately (trunk-based).
- **D-10:** **arduis never deletes anything from disk.** The UI only enables **closing a
  repository** (killing that repo's terminal process groups within the task). The user
  navigates in their own terminal and deletes worktrees/task folders manually. The full
  "conclude task" flow (safe ordered teardown + review) remains Phase 8.

### Lifecycle & sidebar
- **D-11:** On startup, arduis **scans `../<root>-tasks/`** and lists each task found as
  **HIBERNATED** (select/resume relaunches default terminals). Disk is the source of truth —
  no persisted state file.
- **D-12:** The pinned row becomes **`<project> · zsh`**: one terminal at the PROJECT ROOT
  (where trunk compose/tooling runs). A regular workspace, no special-case code (03.1 D-07
  re-targeted).
- **D-13:** Mixed branch state on task creation is resolved **per repo**: branch exists (and
  is not checked out elsewhere) → reuse it; doesn't exist → create from that repo's default
  branch (Phase-2 inference applied repo-by-repo). Checked out in another worktree → abort
  that repo with a clear message; never `--force`.
- **D-14:** The active cap (RAM-03) counts **active (non-hibernated) TASKS**, regardless of
  how many repos/terminals each has (03.1 D-11 re-targeted). A fat task's cost shows in its
  row's RAM figure.

### Carried forward (03.1, re-targeted to task — not re-asked)
- Hibernate kills ALL the task's process groups across all repos (SIGHUP→SIGKILL), keeps
  directories; resume relaunches the default layout (03.1 D-08/D-09).
- Sidebar row RAM = sum of all terminal process groups of all the task's repos; footer
  aggregates across tasks (03.1 D-10).
- tmux model: panes = terminals, windows = tasks; C-Space h/j/k/l within the workspace,
  n/p/number switches tasks (03.1 D-06).
- Branch already checked out → focus/abort, never `--force` (Phase 2).

### Claude's Discretion
- How Task/Project are modeled on the GTK-free `SessionStore` (task with N repos × N
  terminals) — must stay GTK-free and serializable; migration from `WorktreeSession`.
- The "New task" dialog design (repo multi-pick + branch type-or-pick).
- What validates a directory under `<root>-tasks/` as a rediscoverable task during the
  startup scan.
- Symlink mechanics (relative vs absolute), branch-name sanitization for the task dir.
- How "close a repository" is presented in the UI (row menu, pane header, etc.).
- Widget strategy for the project topbar (name display only).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/arduis/session.py` (`SessionStore`, `WorktreeSession`, `TerminalRecord`,
  `default_terminals`, `hibernate_fields`) — extend to model a TASK owning N repos, each with
  its terminals; stays GTK-free/serializable.
- `src/arduis/worktree.py` — git argv builders (`argv_worktree_add_new/existing`,
  `parse_worktrees`, `infer_new_vs_existing`, `sanitize_branch_for_dir`,
  `branch_checked_out_path`) — reuse per member repo; `worktree_dir_for` needs a task-folder
  variant (`<root>-tasks/<branch>/<repo-name>`).
- `src/arduis/layout.py` (binary split tree, per-workspace instance) — unchanged; a task's
  workspace just starts with 2×N leaves (D-01/D-02).
- `src/arduis/resource_monitor.py` — already walks process groups; sum across all repos'
  terminals per task.
- `src/arduis/keymap.py` + the C-Space prefix machine in `window.py` — unchanged mechanism;
  workspace-switch targets become tasks.

### Established Patterns
- Logic in GTK-free modules with `window.py` as a thin widget reflection
  (`_reflect_layout`/`_build_widget`) — keep the project/task model GTK-free, UI as a view.
- Async git via `run_git_async` (`git_service.py`) chained callbacks — task creation is a
  sequence of per-repo worktree adds off the GTK loop.
- Teardown via `_teardown_pgid` (SIGHUP→SIGKILL per pgid) — iterate every terminal of every
  repo of a task.

### Integration Points
- `window.py:_resolve_repo_root` (cwd → `git rev-parse`) becomes project-root resolution
  (D-05/D-07: scan subdirs for `.git` first, fall back to single-repo).
- `window.py:_open_and_add` / `_create_worktree` / `_resolve_base_then_add` — become per-repo
  loops materializing the task folder + symlinks before spawning terminals.
- `window.py:_rebuild_sidebar` / `_swap_workspace` / `_on_hibernate` / `_on_resume` /
  `_poll_ram` — re-target row unit from worktree-session to task.
- Headerbar (`Adw.ToolbarView`/`Adw.HeaderBar` in `window.py`) — hosts the project-name
  topbar (D-06).

<specifics>
## Specific Ideas

- "Não leve como regra repo = coluna/shell — mas pode ser assim por default": the default
  layout is a convenience, never a structural constraint.
- "Eu deleto manual tudo; na tela só deve habilitar fechar um repositório, aí eu navego pelo
  meu terminal e deleto" — deletion stays in the user's hands; arduis only kills processes.
- tmux mental model remains the anchor: panes = terminals, windows = tasks.
- Real layout this serves: a root like `livon/` with `backend/`, `frontend/`, `keycloak/` +
  root `CLAUDE.md`/`docker-compose.yml`; a feature crosses repos, so the task (not the repo)
  is the unit — otherwise agents of one feature fragment across tabs and "who's waiting"
  breaks.

</specifics>
