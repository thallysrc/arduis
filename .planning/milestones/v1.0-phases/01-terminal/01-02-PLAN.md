<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| arduis window → host PTY child (`zsh`) | Direct native PTY spawn of the user's login+interactive shell with full user privileges; no sandbox in v1. |
| Window lifecycle → host process group | Window close must reliably terminate the child process group; failure leaks long-lived host processes (orphans). |
| GTK `child-exited` raw status → app | Untrusted-shaped integer (raw waitpid status) crosses into the app and must be decoded, not trusted as an exit code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Denial of Service (resource leak) | `_on_close_request` teardown | mitigate | SIGHUP the child PGID via `os.getpgid(pid)` then SIGKILL on a 1.5s `GLib.timeout_add`; verified by the no-orphan acceptance step (`pgrep` after close). A2: GTK4 `close-request` (not `delete-event`) so the handler actually fires. |
| T-01-06 | Tampering / Elevation | `Vte.Terminal.spawn_async` argv/envv | mitigate | argv/envv come only from Plan-01's tested `build_spawn_command` (list literals, no shell string, no flatpak prefix); GTK layer does not reassemble them from user input. |
| T-01-07 | Spoofing (visual) | terminal color palette | mitigate | App applies `set_colors`/`set_color_cursor` from its own Dracula constants so the shell cannot impose deceptive colors (D-06/D-07). |
| T-01-08 | Information disclosure | `child-exited` status handling | accept | Status decoded via `decode_exit`; only stored on `self._last_exit` for local use. No PII, single-user local tool — accepted. |
| T-01-09 | Repudiation | n/a (no audit need) | accept | Local single-user dev tool; no audit-log requirement in v1. |
</threat_model>
