<objective>
Build `docker_service.py` — the single new gi-importing SERVICE module for Phase 7: a thin
`run_compose_async(argv, on_done, runner=None)` that wraps the list-form argv (built by the GTK-free
`compose.py` in Wave 1) through the `HostRunner` seam and runs it off the GTK main loop via
`Gio.Subprocess` + `communicate_utf8_async`, handing back `(exit_status, stdout, stderr)` on the
GLib loop. It is a near-verbatim clone of `git_service.run_git_async` — it builds nothing and parses
nothing (compose.py owns argv, window.py owns orchestration).

Purpose: `docker compose up -d` can block for minutes on a first image pull; a synchronous call
would freeze the UI (Pitfall 3). This module is the async boundary that keeps every compose call off
the GTK loop, and the `HostRunner` route is the single home the v2 Flatpak path will hook. It is
deliberately tiny — the only line it adds over git_service is the function name. Wave 3's window
wiring calls it for config/up/down/ls.
Output: `src/arduis/docker_service.py`, `tests/test_docker_service.py`; full suite green
(baseline 274 + Wave-1 new + this).
</objective>

From src/arduis/git_service.py (clone verbatim, rename run_git_async -> run_compose_async):
```python
from __future__ import annotations
import gi
gi.require_version("Gtk", "4.0")
from gi.repository import Gio  # noqa: E402
from arduis.host_runner import HostRunner  # noqa: E402

def run_git_async(argv: list[str], on_done, runner: HostRunner | None = None) -> None:
    wrapped = (runner or HostRunner()).wrap_argv(argv)   # route through the seam
    proc = Gio.Subprocess.new(
        wrapped,
        Gio.SubprocessFlags.STDOUT_PIPE | Gio.SubprocessFlags.STDERR_PIPE,
    )
    def _cb(p, res):
        ok, out, err = p.communicate_utf8_finish(res)
        on_done(p.get_exit_status(), out or "", err or "")
    proc.communicate_utf8_async(None, None, _cb)
```

- **Clone, don't invent:** `docker_service.run_compose_async` is `git_service.run_git_async` with the
  function renamed. Same imports, same `gi.require_version("Gtk","4.0")`, same `Gio.Subprocess.new`

  that it builds/parses nothing (compose.py owns argv), and that argv route through HostRunner for the
  v2 Flatpak seam. Mirror git_service's "Threats" + "Concurrency" docstring notes (T-07-08 below).

  requires gi/Gtk available — the venv has system-site-packages, so this is fine, same as any
  gi-importing test today). To test argv ROUTING without spawning a real process, pass a FAKE runner
  whose `wrap_argv` records the argv and returns a sentinel, and monkeypatch
  `docker_service.Gio.Subprocess.new` to a stub that captures the wrapped argv and returns a fake proc
  whose `communicate_utf8_async` immediately invokes the callback with a canned `(True, "out", "")`.
  Assert: `wrap_argv` was called with the exact argv; `Gio.Subprocess.new` got the wrapped argv with
  STDOUT_PIPE|STDERR_PIPE; `on_done` fired with the decoded `(0, "out", "")`. This pins the wiring;
  the REAL subprocess path is exercised by Plan 05's host smoke/UAT.
  If monkeypatching `Gio.Subprocess.new` proves awkward, the MINIMUM acceptable test asserts the fake
  runner's `wrap_argv` received the exact argv (routing through the seam) — the on_done shape is then
  covered by reading the (cloned, identical) git_service test for run_git_async.
- **No new dependency.** stdlib + gi only (already present).
