# Phase 03.4 Plan 02: Delete Wrong-Level Chips + Cap-Union Lock Summary

Removed the 03.3 per-repo chip machinery (the wrong level — the topbar holds PROJECTS, not
repos), reverted the New-task dialog to all-member-repos-checked, locked the global cap as a
UNION across all projects with a GTK-free test, and repaired the stale `docs/MOTIVATION.md`
line. The singleton model in `window.py` is untouched — `main` stays launchable single-project,
just chip-free; the registry swap is Plan 03.

## What Was Built

- **ChipState + all chip render/reflect machinery removed (D-04).** Deleted from `window.py`:
  the `from arduis.topbar import ChipState` import, the three chip-map fields in `__init__`
  (`_chip_state` / `_chip_btn_by_repo` / `_chip_dot_by_repo`), the six methods
  (`_build_chip_bar`, `_make_chip`, `_make_chip_toggled_cb`, `_on_toggle_chip`, `_restyle_chips`,
  `_reflect_active_chips`), the `_build_chip_bar()` call at the end of `_resolve_project`, the
  three `_reflect_active_chips(...)` callers (`_swap_workspace`, `_show_hibernated_placeholder`,
  the create path), and the `win.toggle_chip` `Gio.SimpleAction` registration. `topbar.py` is
  emptied to a superseded-placeholder docstring (the `ChipState` class is gone). `grep -rn
  'ChipState' src/arduis/` prints CLEAN package-wide.
- **`_chip_bar` Box retained as an empty placeholder slot.** The headerbar `pack_start` Box stays
  (no children, no build call) with a comment that Plan 03 reuses this exact slot for the project
  tabs switcher (D-03/D-04). The `.arduis-chip*` CSS and `_MAX_VISIBLE_CHIPS` are intentionally
  kept for Plan 03 reuse (active-tab styling + `+N` overflow pattern).
- **New-task dialog reverted to all-checked (D-12).** The ChipState-seeded `default` branch is
  gone; the per-repo CheckButtons now seed straight from `default = set(self._member_repos)` —
  every member repo checked by default, still user-overridable per task. Repo names still reach
  the widget via `Gtk.CheckButton(label=...)` (plain text, no `set_markup`) — the T-03-09 /
  T-03.4-05 label-injection guard is preserved.
- **Cap-union behavior locked at the GTK-free level (D-09).** Added two tests to `test_caps.py`
  over the existing (unchanged) `caps.py`: `test_cap_counts_union_across_projects` builds two
  `Project`s (4 ACTIVE in A, 3 in B), asserts `active_count(union) == 7` and `at_cap(union) is
  True` while neither single store trips the cap alone (Pitfall 4 regression guard); and
  `test_cap_union_hibernated_excluded` confirms a HIBERNATED task in either store is not counted.
- **`docs/MOTIVATION.md` repaired.** The stale pane-layout bullet ("Repos no topbar, worktrees na
  sidebar") now agrees with the approved 3-level model: topbar = switchable multi-repo projects
  ("ambos vivos"), sidebar = the selected project's tasks, member repos surface only in the "Nova
  task" dialog.

## Threat Mitigations Applied

- **T-03.4-05** (Tampering, New-task dialog repo labels): member-repo names stay on
  `Gtk.CheckButton(label=...)` (plain text widget, never `set_markup`) — the T-03-09
  label-injection guard is preserved through the dialog revert.
- **T-03.4-06** (resource leak, global cap underflow): the cap-union test (D-09) locks that the
  cap counts the UNION of all projects' tasks, so agents can't silently exceed the machine-global
  limit by splitting across projects (Pitfall 4).

No new threat surface introduced (pure deletion + a GTK-free test + a docs edit).
