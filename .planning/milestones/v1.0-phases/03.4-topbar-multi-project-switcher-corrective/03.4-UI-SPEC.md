## Surface Scope (this contract covers ONLY)

The topbar **project switcher** that replaces the 03.3 repo chip bar in the `Adw.HeaderBar`:

1. The project tab row (one tab per open project, basename label, active-tab visual state).
2. The trailing **"+ Abrir projeto"** affordance (folder-picker trigger).
3. Overflow behavior when tabs exceed header width (reuse the 03.3 `+N` `Gtk.MenuButton` pattern).
4. Empty/degenerate states (exactly one project; active vs background tab styling).
5. Project removal affordance + the confirm-on-live-tasks `Adw.AlertDialog` (D-10).

**Explicitly OUT of scope** (do NOT design): the New-task dialog repo picker (already exists,
D-12 just routes member repos there); Command/Swarm/Review room tabs (v2); per-project status
rollup dot on the tab (DEFERRED per CONTEXT â€” tabs are **label-only**); drag-reorder / pinning /
per-project themes (deferred).

## Spacing Scale

Declared values (multiples of 4; matches the existing `_build_css` and headerbar spacing). The
switcher MUST reuse these â€” do not introduce new spacing tokens.

| Token | Value | Usage in this surface |
|-------|-------|------------------------|
| xs | 4px | `.arduis-chip-bar` horizontal padding (`padding: 0 4px`) â€” reused for the tab bar |
| sm | 6px | Inter-tab gap + intra-tab box spacing (existing `_chip_bar` Box `spacing=6`, `_make_chip` box `spacing=6`) |
| sm2 | 8px | Tab horizontal padding (`.arduis-chip`/tab `padding: 2px 8px`) |
| â€” | 2px | Tab vertical padding (existing chip vertical padding; keeps tabs compact in the 1-row header) |
| md | 16px | Pane-header / hint-bar / sidebar inner padding (context only â€” not the switcher) |

Exceptions: tab vertical padding is **2px** (not on the 4-grid) â€” inherited verbatim from the
existing `.arduis-chip` rule so the switcher sits at the same height as today's chip row inside
the single-row `Adw.HeaderBar`. This is a pre-existing, intentionally-compact header metric, not
a new exception introduced by this phase.

## Typography

GTK does not use a web type scale; sizes come from CSS `font-size` in `_build_css` plus the GTK
default UI font. The switcher reuses the existing header/label sizes â€” declare no new ones.

| Role | Size | Weight | Usage in this surface |
|------|------|--------|------------------------|
| Tab label (project basename) | system default (~14px) | 400 (regular) for background tabs | One `Gtk.Label` per tab via `set_text` (NEVER `set_markup` â€” T-03-09 path-injection guard) |
| Tab label â€” active | system default (~14px) | 600 (semibold) | Active project tab uses weight 600 to read as "selected" alongside the border accent |
| Overflow / "+ Abrir projeto" button label | system default (~14px) | 400 | `Gtk.MenuButton` / `Gtk.Button` flat label |
| Sub-labels (row branch / subline) | 13px / 11px | 600 / 400 | Context only (sidebar rows) â€” NOT the switcher |

Weights used in this surface: **exactly two** â€” 400 (regular, background tabs) and 600 (semibold,
active tab). Line height: GTK single-line label default (no multi-line text in the switcher).

## Color

All values resolve from the active `Theme` via CSS classes â€” the switcher widgets attach classes,
they do **not** set hex. Reference hexes are the **Dracula default** field values (orientation
only). This preserves UI-02 theme-swapping (Nord / Solarized / Gruvbox all re-skin the switcher
for free).

| Role | Theme field / CSS class | Dracula ref | Usage in this surface |
|------|--------------------------|-------------|------------------------|
| Dominant (60%) | window background (GTK default dark base, `Adw.ColorScheme.FORCE_DARK`) | `#282a36` | Canvas + window chrome behind the header |
| Secondary (30%) | `theme.surface` â†’ `.arduis-sidebar` / header surface | `#21222c` | Headerbar background the tab row sits on; active-tab fill |
| Accent (10%) | `theme.accent` â†’ `.arduis-chip-active` border (reuse/rename to `.arduis-tab-active`) | `#bd93f9` | **Active project tab ONLY**: 1px accent border. Reserved â€” never applied to background tabs, the "+ Abrir projeto" button, or the overflow `+N` button |
| Destructive | `Adw.ResponseAppearance.DESTRUCTIVE` (libadwaita system red) | (system) | The "Remover" confirm response in the live-tasks `Adw.AlertDialog` (D-10) ONLY |

Accent reserved for: **the active project tab's 1px border + semibold label** (the single
"which project am I in" signal). Explicitly NOT for: background/inactive tabs, the "+ Abrir
projeto" button, the `+N` overflow button, hover states (GTK provides the default flat-button
hover; no accent on hover).

**CSS class decision (executor):** reuse the existing `.arduis-chip` / `.arduis-chip-active`
classes for tabs, OR rename to `.arduis-tab` / `.arduis-tab-active` with identical rule bodies.
Either way the **active-tab rule body is unchanged** from the existing `.arduis-chip-active`
(`border: 1px solid {theme.accent}; background-color: {theme.surface};`). Do NOT delete the
`.arduis-chip*` rules and inline new hexes â€” keep them theme-sourced. The per-repo
`.arduis-dot-*` status-dot classes **survive** (used in the New-task dialog / sidebar) but are
**NOT rendered on project tabs** (per-project status rollup deferred â€” tabs are label-only).

## Interaction & State Contract

| State | Visual / behavior |
|-------|-------------------|
| **Active tab** | `.arduis-chip-active` (1px `theme.accent` border + `theme.surface` fill) + semibold (600) label. Exactly one active at all times (linked `ToggleButton` group). |
| **Background tab** | flat, regular (400) label, no accent border. Clickable to switch (re-points the active project + runs the existing workspace swap). Background project's terminals stay alive (D-08) â€” no visual "asleep" treatment in this phase. |
| **Hover** | GTK default flat-button hover highlight (no accent). |
| **Exactly one project (degenerate / launch-dir auto-register, D-07)** | Render the single tab as the active tab (it reads as "the current project", not a lone switcher demanding a choice). The "+ Abrir projeto" affordance still shows so the user can add a second project. Do NOT show overflow or any "switcher" chrome beyond the one tab + the add button â€” must look clean, like today's single-project launch. |
| **Overflow (tabs > header width / `_MAX_VISIBLE`)** | First `_MAX_VISIBLE` tabs inline, remainder folded into a `+N` `Gtk.MenuButton`; no horizontal scroll (matches the 03.3 rejection of scroll for VTE-focus reasons). |
| **Switching tabs** | Re-point active project, rebuild sidebar from that project's store, run existing `_swap_workspace`; re-grab focus on the swapped-in terminal (Pitfall 2). No teardown of the previous project. |
| **Removing a project (no live tasks)** | Drop from the remembered list + `projects.json`, remove the tab; no dialog. Never deletes worktrees/dirs on disk. |
| **Removing a project (live/running tasks, D-10)** | Show the DESTRUCTIVE confirm `Adw.AlertDialog` first; on confirm, tear down (kill pgids + `compose down -v`) then drop from list + JSON. Cancel is the default response. |
| **Missing remembered root on load (D-06)** | Skip silently and rewrite `projects.json` to drop it (no broken/greyed tab this phase â€” keep-greyed is a deferred nicety). |

## Accessibility / Input Notes

- Single-row `Adw.HeaderBar`: tabs must stay compact (2px vertical padding) so the row height is
  unchanged from the existing chip bar.
- Project names are user/filesystem-derived â†’ always `set_text`, never `set_markup` (label
  injection guard, T-03-09 / V5 Input Validation from RESEARCH).
- Linked `Gtk.ToggleButton` group guarantees exactly-one-active without manual bookkeeping
  (keyboard arrow traversal within the linked group comes free from GTK).
- Tooltips ("Abrir projeto", "Mais projetos", project full `root` path on each tab) give the
  hover affordance GTK conventions expect.
