<plan_decisions>
- **Consent flow (D-02):** dialog shown iff `not is_installed(settings, path)` AND no
  declined-marker. "Instalar" → backup `~/.claude/settings.json.arduis-backup` (copy, only if
  a settings file exists) + atomic write (tmp + os.replace in the same dir) of
  `merged_settings(...)`; missing settings file → treated as `{}` and created. "Agora não" →
  touch marker, `self._degraded = True`. Settings read/write is small and startup-only —
  synchronous stdlib I/O is acceptable here (one-shot, < ms; not a long job).
- **Dot colors (D-06):** waiting `#ffb86c` (orange), running reuses `_DOT_ACTIVE` `#50fa7b`,
  ready `#8be9fd` (cyan), idle `#7a9e7e` (muted grey-green — D-06 "grey-green"), ended +
  hibernated stay `#6272a4`. New CSS classes: `arduis-dot-waiting`, `arduis-dot-ready`,
  `arduis-dot-idle`.
- **Pane-header indicator (D-07):** a small "●" Gtk.Label inserted before the badge in
  `_make_leaf`, visible only for task AGENT terminals (hidden for shells/pinned main);
  handle kept in `self._pane_dot_by_tid`.
- **Notification identity (D-09):** ONE `Notify.Notification` object kept per terminal
  (`self._notif_by_tid`); subsequent waitings call `.update(...)` + `.show()` on the same
  object → server-side replacement, no stacking. App name "arduis" via `Notify.init("arduis")`.
- **Message escaping:** notification body runs through `GLib.markup_escape_text` (some
  notification servers parse body markup — the message text comes from a state file).
</plan_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| arduis → ~/.claude/settings.json | arduis WRITES into the user's heavily-customized config |
| state-file content → notification body / UI labels | hook-written text rendered by the notification server |
| status dir events → main-loop handlers | inotify events drive code on the GTK loop |
| consent UX → user trust | writing without consent would betray D-02 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | Tampering | settings.json write corrupting the user's file | mitigate | Backup copy before first write; tmp + os.replace atomic write; merge via the Plan-02 tested additive fn; UNPARSEABLE settings → never write at all (degraded mode) |
| T-04-13 | Spoofing/Repudiation | writing hooks without consent | mitigate | Adw.AlertDialog gate; declined marker persisted; is_installed short-circuit makes re-runs silent no-ops |
| T-04-14 | Tampering | notification body markup injection from state-file message | mitigate | GLib.markup_escape_text on the body; title is the branch name rendered as plain text by the server (and already constrained by branch sanitization upstream) |
| T-04-15 | DoS | monitor event storm / unreadable file crashing the loop | mitigate | Handler is O(1) dict lookup + tolerant read_state (None on failure); atomic renames yield ~2 events/write at human-driven rates |
| T-04-16 | EoP/Tampering | state-file deletion generalizing into user-data deletion (D-10) | mitigate | `_clear_task_state_files` computes paths ONLY via attention.state_file_path under the status dir; no rm of anything else; verification grep below |
| T-04-17 | DoS | os.killpg(pgid, 0) probe on a reused pgid | accept | Signal 0 sends nothing; worst case is a stale-status misread until the next event; same probe already used by Phase-3 RAM machinery |
</threat_model>
